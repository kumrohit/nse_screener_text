## 1. Purpose and scope

A screener over the Nifty 500 universe that accepts filters written in plain English ("stocks taking support at 50 EMA and in an uptrend"), compiles them into a deterministic filter specification, and evaluates that specification against five years of daily price and volume history. Version scope is deliberately limited to price/volume technical analysis: no fundamentals, no intraday, no derivatives data. The parser is required to refuse anything outside this scope rather than approximate it.

## 2. Design principles

The system is built around one central rule: **natural language never touches computation.** The LLM's sole responsibility is translating text into a validated JSON specification (the DSL). Everything downstream — indicators, condition evaluation, result assembly — is deterministic, pure, and unit-testable. Consequences of this rule:

1. Every screen is reproducible: the compiled DSL plus the data snapshot date fully determine the output.
2. Ambiguity is resolved once, in a documented canonical vocabulary (§7), not per-query by model whim.
3. The user always sees the compiled interpretation in plain English before results, and can bypass the LLM entirely with `--json`.
4. The parser must return an explicit error for unmappable phrases. Guessing is a defect, and the golden-query suite (§10) treats it as one.

Secondary principles: fail loud (stale data, unknown fields, and missing benchmarks raise, never silently return empty results); no look-ahead anywhere (pivots need confirmation bars, historical as-of screens only use data up to the as-of row); prefer auditable pure-pandas formulas over indicator libraries.

## 3. Architecture

```
 user text ──► parser.py (LLM, canonical vocab) ──► DSL JSON
                                                       │
                                             dsl.py validate + describe
                                                       │ echoed to user
 yfinance/NSE ──► data_ingest.py ──► prices.parquet   │
                                          │            ▼
                                   indicators.py ──► evaluator.py ──► results table
                                   (daily panels,     (condition        (CLI print /
                                    weekly panels,     semantics)        CSV export)
                                    sr.py levels)
```

Module inventory: `config.py` (all tunables), `universe.py` (constituents), `data_ingest.py` (prices + benchmark), `indicators.py` (daily and weekly panels), `sr.py` (swing support/resistance), `dsl.py` (schema, validation, English echo), `evaluator.py` (condition semantics, screen runner), `parser.py` (LLM translation), `cli.py` (backfill/update/screen commands).

## 4. Data layer

**Universe.** Official Nifty 500 constituent CSV from NSE archives, fetched with browser-like headers (NSE blocks default user agents and most datacenter IPs), cached to `data/nifty500.csv`. On fetch failure the cached copy is used with a warning; with no cache the system refuses to run and tells the user how to obtain the file manually. The universe drifts with semi-annual index rebalances — re-fetch with `force_refresh=True` quarterly.

**Prices.** yfinance daily bars with `.NS` suffix, five years of history, `auto_adjust=True`. Auto-adjustment folds in splits and bonuses (essential — unadjusted data destroys every moving average at each corporate action) and also dividends (a minor distortion of raw levels, acceptable for technical work and strictly better than the alternative). Downloads run in chunks of 50 tickers with three retries and exponential-ish backoff, plus a two-second inter-chunk sleep, because yfinance throttling is the dominant operational risk of the free-data route. Storage is long-format Parquet (`symbol, date, open, high, low, close, volume`), roughly 600k rows for the full universe.

**Cleaning.** Rows are dropped when any OHLC value is non-positive or high < low. Timestamps are normalised to tz-naive dates.

**Incremental updates.** `update` re-fetches a ten-day tail for all symbols and merges with `keep="last"`, absorbing Yahoo's occasional late corrections. Schedule after ~18:30 IST when NSE close data settles on Yahoo.

**Benchmark.** Nifty 50 (`^NSEI`) close series stored separately in `benchmark.parquet`, refreshed on every backfill/update, consumed only by the `rel_strength` condition (which fails loud if the benchmark is absent).

**Integrity gates before any screen runs:** staleness check (latest bar within 5 calendar days, else refuse with instructions), minimum history (symbols with fewer than 60 bars are excluded from panels), and a liquidity gate (20-day median turnover ≥ ₹0.5 cr — mostly redundant for Nifty 500 but guards against data glitches surfacing dead tickers).

**Known data risks and mitigations.** (a) NSE symbol ≠ Yahoo ticker for a handful of names post merger/rename; the ingester skips failures silently, so compare the backfilled symbol count against 500 after the first run and maintain an alias map in `universe.py` if needed. (b) yfinance API changes break periodically; the ingest module is the designated swap point — replacing it with NSE bhavcopy ingestion touches nothing else. (c) Survivorship: the store only contains current constituents, so historical as-of screens are biased toward survivors. Acceptable for a discovery tool; documented so nobody mistakes Phase 3 screen-backtests for survivorship-clean research. (d) yfinance adjustment-splice bug: for at least three symbols (CGCL, GPIL, MOTILALOFS), `auto_adjust=True` applies the correct split/bonus ratio but splices it in at the wrong date — a large jump appears on 2024-01-01 for all three, months before their true record dates (Mar/Jun/Oct 2024). Confirmed via `verify --jumps`: `refetch` returns byte-identical data, so this is upstream in Yahoo's own adjusted series, not a local staleness issue. Harmless for live screens once the affected window ages out of the longest lookback (252 bars), but historical as-of replays dated within the corrupted window for those symbols will show wrong EMA/RSI/52-week values. No general fix; flagged here so it isn't mistaken for a fixable ingestion bug.

## 4a. Data layer v2 — NSE bhavcopy (ROADMAP Item 3, pre-cutover)

`screener/bhavcopy.py` runs **side-by-side** with the yfinance store above — its own store (`data/bhavcopy_prices.parquet`), read by nothing in the evaluator/webapp/CLI screen path yet. Cutover happens only after ~2 weeks of evidence from the cross-source check below.

**Source.** NSE's daily `sec_bhavdata_full_DDMMYYYY.csv` — confirmed by fetching a live file (2026-07-03) rather than assumed from training data: one file has OHLCV *and* delivery % together (`SYMBOL, SERIES, DATE1, ..., CLOSE_PRICE, ..., TTL_TRD_QNTY, ..., DELIV_QTY, DELIV_PER`), filtered to `SERIES == "EQ"`. A separate delivery-only file turned out to be unnecessary. `bhavcopy.update_bhavcopy_store()` fetches missing business days since the store's last date (or the last 10 calendar days on first run); a 404 is treated as a weekend/holiday, not a failure — NSE's own calendar is the source of truth. First real run (2026-07-05) pulled 2,431 symbols (all NSE EQ series, not just Nifty 500), 14,403 rows, 2026-06-25 → 2026-07-03.

**Acceptance check, run for real:** bhavcopy closes vs the yfinance store for the same date (2026-07-03), all 500 Nifty 500 symbols — median absolute difference ~2×10⁻⁶%, max ~5×10⁻⁶% (floating-point rounding only), zero symbols over 0.5%. Both sources are unadjusted for that single day (adjustment convention only affects *historical* bars), so this is a clean apples-to-apples check.

**Prices are raw/unadjusted** from both NSE files — same as yfinance without `auto_adjust`. Our own adjustment pipeline: `fetch_corporate_actions()` hits `/api/corporates-corporateActions` (needs a cookie warm-up `GET` to nseindia.com first; that warm-up itself may 403 yet still set the cookies the API call needs, so its status isn't checked). `parse_adjustment_factor(subject)` extracts a multiplicative factor from the free-text `subject` line using two regexes built from **real** NSE text (fetched 2026-07-05, Jan 2024–Jul 2026 window), not guessed formats:
- `"Bonus X:Y"` (X new shares per Y held) → `factor = Y / (X + Y)`
- `"Face Value Split (Sub-Division) - From RsA/- Per Share To RsB/- Per Share"` → `factor = B / A`

Splits/bonuses only — dividends and anything else return `None` and are excluded from adjustment (documented divergence from yfinance's `auto_adjust`, which folds dividends in too). CGCL is a good cross-check: NSE lists *two* separate 2024 actions for it (a 1:1 bonus **and** a 2→1 face-value split), and `0.5 × 0.5 = 0.25` matches the real jump ratio investigated in ROADMAP Item 0 almost exactly. `build_adjustment_factors()` compounds each symbol's factors going backward in time (a bar's cumulative factor is the product of its own action's factor and every *later* action's factor — the same convention `auto_adjust` uses); `apply_adjustments()` must apply actions **latest-ex-date-first**, since an earlier action's `date < ex_date` mask is always a subset of every later action's mask — applying oldest-first would let the later (broader, single-factor) write clobber the earlier (narrower, correctly-larger cumulative-factor) one. Caught and fixed during implementation; regression-tested (`TestAdjustmentCompounding.test_apply_adjustments_ordering`).

**Cross-source consistency check** (`verify.check_cross_source`, folded into `python -m screener.cli verify` automatically): compares overlapping `(symbol, date)` closes between the two stores and WARNs (not FAILs — bhavcopy isn't consumed by any screen yet) above a 0.5% threshold. First real run found 45/3,000 overlapping bars over threshold, concentrated in 11 symbols (e.g. UNIONBANK ~2.95%, TECHM ~2.60%) — but the gap is *constant* across every date for a given symbol, the signature of a past dividend adjustment already folded into yfinance's series (not a bug; exactly the divergence source documented in §4(d) above). A systematic, non-constant gap would be the signal to actually investigate before counting a day toward the 2-week cutover evidence window.

**Not yet done** (explicitly gated): the `delivery` DSL condition, its parser vocabulary, and the "accumulation" preset — ROADMAP marks these "only after cutover," so they aren't wired into `dsl.py`/`evaluator.py`/`parser.py` yet even though `delivery_pct` already exists in the bhavcopy store. Also not done: the config-flag cutover itself, and adding `bhavcopy-update` to the nightly cron (currently a manual command — add it alongside `update` once the side-by-side evidence period is underway).

## 5. Indicator engine

Pure pandas/numpy; every formula lives in `indicators.py` and is auditable. Wilder-style indicators (RSI, ATR, ADX) use `ewm(alpha=1/n)` smoothing, matching standard charting-platform values. EMAs use `span=n, adjust=False` with `min_periods=n` so warm-up rows are NaN rather than misleading — and every condition treats NaN as False.

Daily panel columns: OHLCV; EMA 10/20/50/100/200 each with a 5-bar difference slope; SMA 20/50/150/200 each with a 5-bar difference slope (150 and its slope added in v0.9 for the Minervini "stage 2" template, §12c); RSI(14); ATR(14) and ATR%; ADX/+DI/−DI(14); MACD(12,26,9) line/signal/histogram; Bollinger(20, 2σ) upper/lower/width%; 20-day average volume and volume ratio; turnover in ₹ crore; 252-bar rolling 52-week high/low and percentage distances; rate of change over 5/21/63/126/252 bars (≈ 1 week / 1 month / 1 quarter / 6 months / 12 months); `mom_12_1` — the 12-1 skip-month momentum return from 252 bars ago to 21 bars ago (Jegadeesh & Titman 1993 convention, §12c) — deliberately distinct from `roc_252`, which includes the most recent month.

Weekly panel (computed lazily, only when a screen contains a weekly condition): W-FRI resample of OHLCV; EMA 10/20/40 with 3-week slopes; RSI(14); ROC over 4 and 13 weeks. The in-progress week appears as a partial bar — a deliberate choice so "weekly uptrend" reflects the current state rather than last Friday's, at the cost of the last weekly bar being mutable until Friday's close.

Trend is defined canonically, not left to interpretation. Daily uptrend: close > EMA50 > EMA200 and EMA50 5-bar slope > 0. Weekly uptrend: close > EMA20w > EMA40w and EMA20w 3-week slope > 0. Downtrends are exact mirrors.

## 6. Filter DSL

A screen is `{"logic": "AND"|"OR", "conditions": [...], "as_of": "latest"|ISO-date}`. Validation is strict: unknown condition types, unknown fields, bad operators, and weekly-timeframe use of daily-only fields all raise `DSLValidationError` before any data is touched. Twenty-one condition types:

| Type | Keys (defaults) | Semantics |
|---|---|---|
| `compare` | left, op, right | Field vs field-or-number at the as-of row. |
| `proximity` | target, ref, tolerance_pct, lookback (3) | \|target−ref\|/ref ≤ tol on any bar in the window. |
| `trend` | direction | Canonical trend definition (§5). |
| `support_at_ma` | ma, tolerance_pct (1.5), lookback (3) | See below. |
| `cross` | fast, slow, direction, lookback (3) | Sign change of fast−slow within the window. |
| `volume_spike` | min_ratio | volume / 20-day average ≥ ratio. |
| `range` | field, min and/or max | Inclusive band check. |
| `change` | field, window, op, value_pct | Percent change over window bars. |
| `near_support` | tolerance_pct | Close within tol above nearest swing support (§8). |
| `near_resistance` | tolerance_pct | Close within tol below nearest swing resistance. |
| `breakout_resistance` | lookback (5), buffer_pct (0) | See §8. |
| `rel_strength` | window, op, value_pct | Stock return − Nifty return over window, in pct points. |
| `candle` | pattern, lookback (1) | One of 6 exact candlestick formulas (§9a) within the window. |
| `tight_range` | max_range_pct, bars (10) | N-bar high-low span as % of the window low. |
| `bb_squeeze` | percentile (20), lookback (252) | Bollinger bandwidth vs its own trailing distribution. |
| `flat_base` | bars (20), max_range_pct (12), max_from_52w_high_pct (15) | Tight range near the 52-week high (§9a). |
| `sector` | in (list of exact industry strings) | Stock's Nifty 500 industry classification is one of the given sectors. |
| `rs_percentile` | window (63), basis ("return"), op, value | Stock ranked as a percentile (0–100) across the universe on the as-of date (§6a). `basis="return"`: own window-bar return. `basis="mom_12_1"`: the 12-1 skip-month construction (§12c); `window` is ignored. |
| `sector_rank` | window (63), top **xor** bottom | Stock's sector is among the top/bottom N sectors by equal-weight window-bar momentum (§6a). |
| `gap` | direction, min_gap_pct (2.0), lookback (3) | Open vs the prior bar's close, on any bar within the window. |
| `atr_pct_percentile` | op, value | Stock's ATR% (volatility) ranked as a percentile (0–100) across the universe; 0 = least volatile (§12c). |

`compare`, `range`, `trend`, `change`, and `cross` accept `"timeframe": "weekly"`, which routes evaluation to the weekly panel with its restricted field set.

### 6a. Cross-sectional pre-pass

`rs_percentile`, `sector_rank`, and (v0.9) `atr_pct_percentile` need context beyond a single symbol's panel, so `run_screen` (and webapp's `/api/screen`) compute a **cross-sectional pre-pass** once per screen — not once per symbol — before the per-symbol loop: `cross_section.build_cross_section(panels, universe, as_of, window)` returns a table indexed by symbol with that window's return, its RS percentile, the equal-weight mean return of its sector, the sector's momentum rank (1 = best), the panel's `mom_12_1` value and its percentile (v0.9, §12c), and the panel's `atr_pct` value and its percentile (v0.9, §12c — ranked ascending, so 0 = least volatile in the universe on that date). This is the one structural change Item 2 required: `evaluate_symbol`/`explain_symbol` gained `symbol`, `sector_by_symbol`, and `cross_section` parameters (all optional, default `None`, so every existing call site keeps working unchanged). `atr_pct_percentile` and `mom_12_1`-basis `rs_percentile` share the same window-keyed cache as `rs_percentile`/`sector_rank` even though their ranking doesn't depend on the window value — cheap to compute redundantly, and it lets a screen combining them at the same window reuse one cache entry.

Design points: pure function, in-process cache only keyed by `(id(panels), as_of, window)` — no disk state, cheap enough (single-digit milliseconds over 500 symbols × 5y; measured well under the 5s budget) that a persistent cache isn't worth the invalidation complexity in v1. Thin-history symbols (fewer than `window` bars up to `as_of`) get `ret_pct = NaN` and are excluded from `rank(pct=True)` rather than defaulted to the 0th/100th percentile — pandas keeps NaN as NaN through ranking, so this falls out of the implementation rather than needing a special case. Historical as-of correctness: every value is recomputed from the row at or before `as_of`, never sliced from "latest" — verified by a dedicated look-ahead test (same spirit as the pivot-confirmation test in §8): two symbols whose relative sector performance flips between an early date and today rank in opposite order at each date.

`sector_rank`'s `bottom` (the mirror of `top`, for "laggard" screens) is a controlled extension beyond the two DSL condition shapes literally spec'd for Item 2 — `top`/`bottom` are mutually exclusive, validated the same way `trend`'s up/down and `cross`'s above/below are.

**"Taking support at a moving average"** — the phrase that motivated the whole design — is defined as: within the lookback window, (a) the low touched the MA (came within tolerance, or pierced below it), (b) the close held at or above MA × (1 − tol) on *every* bar of the window, and (c) the latest close is strictly above the MA. Condition (b) is what separates a bounce from a breakdown: a stock that slices through the MA and closes 3% below it has touched the MA but not taken support. Condition (c) requires the bounce to have actually happened, excluding stocks still sitting on the level.

**Tolerance calibration (2026-07-05):** ran `support_50ema_uptrend` across four as-of dates (2026-04-01, 05-01, 06-01, 07-03), 3 matches each. Decision: keep the 1.5% / 3-bar default. Revisit if a larger sample or the near-miss lists suggest it's too tight or too loose in live use.

The `describe()` function renders any validated spec back to plain English; the CLI prints this before every run and `--dry-run` stops there. This is the primary trust mechanism.

## 7. NL parser

Single Anthropic API call (`claude-sonnet-4-6`, temperature default) with a system prompt containing the full schema, the exhaustive allowed-field list, and the canonical vocabulary. Key mappings: uptrend/downtrend → `trend`; "taking support at / bouncing off <MA>" → `support_at_ma` (the user naming an MA distinguishes this from horizontal `near_support`); "near X" → `proximity` tol 2.0; golden/death cross → EMA50/EMA200 cross, lookback 5; volume spike → ratio 1.5, "huge/massive" → 2.5; oversold/overbought → RSI ≤30 / ≥70; "near 52-week high" → distance ≥ −5%; "strong trend" → ADX ≥ 25; "outperforming the Nifty" → `rel_strength` 63-bar > 0; "weekly uptrend" → weekly trend condition; "<sector> stocks"/"in the <sector> sector" → `sector` (common short forms like "IT" or "pharma" map to the exact industry string — see §6a — but only when the prompt's fixed sector list has an unambiguous match); "RS above N" → `rs_percentile` window 63, op ≥; "market leaders"/"top sector" → `sector_rank` top 3; "lagging sector" → `sector_rank` bottom 3. Time words map as 1 week = 5 bars, 1 month = 21, 3 months = 63 (weekly: 4 and 13).

Two traps are encoded explicitly in the prompt. First, "breakout above 52-week high" must **not** compile to `close > high_52w` — the rolling high includes today, so that comparison is nearly unsatisfiable; it maps to `pct_from_52w_high ≥ −0.5` instead. Second, bare "moving average" defaults to EMA; "DMA" or "simple" selects SMA.

Hard parser rules: output raw JSON only; never invent thresholds beyond canonical defaults unless the user states them; AND unless the user says or/either; return `{"error": "..."}` for anything unmappable (fundamentals, news, intraday, or a sector adjective that doesn't clearly match one of the fixed industry strings). Parser output still passes through `dsl.validate()` — the LLM is never trusted to be well-formed.

## 8. Swing support/resistance methodology

Pivot detection uses the fractal method: bar j is a pivot high when its high is the maximum of bars [j−k, j+k], k = 5 (mirror for lows). The final k bars of any window can never confirm a pivot because the right side of the fractal hasn't printed — this is the standard confirmation lag and doubles as the look-ahead guard, verified by a dedicated test.

Levels are built per symbol, per as-of row, from pivots in the trailing 250 bars: pivot prices are sorted and greedily clustered (a price joins the running cluster if within 1.0% of the cluster's last member); each cluster's mean is a candidate level with a touch count; levels with fewer than 2 touches are discarded as noise. Nearest support is the highest surviving level at or below the close; nearest resistance is the lowest above it.

`breakout_resistance` computes levels **as of `lookback` bars ago** (so the breakout move itself cannot create or erase the level being broken), takes the nearest resistance from that vantage point, and requires the current close to exceed it by the optional buffer. This is intentionally conservative; the documented limitation is that very recent consolidations (younger than pivot confirmation + lookback) can't produce breakout signals yet.

Parameter choices (k=5, 250-bar lookback, 1% cluster tolerance, 2-touch minimum) are defensible defaults, not optimised values — they live in `sr.py` constants and are candidates for sensitivity analysis in Phase 3.

## 9a. Pattern definitions

Candlestick and consolidation conditions use these exact formulas (bar j; body = |close−open|, range = high−low, wicks measured from body edges). **Inside bar**: high < previous high AND low > previous low. **NR7**: range is the strict minimum of the last 7 bars. **Bullish engulfing**: previous bar red, current bar green, current open ≤ previous close and current close ≥ previous open (bodies only; wicks ignored). **Bearish engulfing**: exact mirror. **Hammer**: lower wick ≥ 2× body AND upper wick ≤ 30% of range. **Shooting star**: mirror. `candle` conditions accept a lookback (default 1 = latest bar only). **Tight range**: (max high − min low) / min low over N bars ≤ threshold. **BB squeeze**: current Bollinger bandwidth ≤ the given percentile of its own trailing 252-bar distribution (≥60 observations required, else fails closed). **Flat base**: tight range over 20 bars (≤12%) AND close within 15% of the 52-week high — the pre-breakout base structure. These definitions are deliberately strict and singular; trader phrases map to them via the canonical vocabulary or not at all.

A curated preset library (`presets.py`, 26 named screens grouped by intent — trend continuation, breakouts, reversals, relative strength, bearish) exposes ready-made specs via `GET /api/presets` (web dropdown), `screener presets`, and `screener screen --preset <id>`. Every preset is validated at import time, so DSL changes that break a preset fail the test suite immediately. Since v0.9 (§12c), every preset also carries an `evidence` object pointing back to `LITERATURE.md`.

## 9. Evaluator semantics

Conditions are pure functions `(panel, condition, row_index) → bool`. NaN anywhere in a condition's inputs yields False (a stock with insufficient history cannot match — it is never accidentally included). `as_of` resolves to the last row at or before the given date, independently for daily and weekly panels. `logic` applies flat AND/OR across conditions; nested boolean trees are explicitly out of scope for v1 (the parser refuses queries requiring them). The runner applies the liquidity gate, evaluates each symbol, and emits a result row of screening context: close, % vs EMA50, RSI, volume ratio, 1M/3M returns, distance from 52-week high, plus name and industry from the universe file, sorted by 3-month return.

## 10. Testing strategy

Three layers. **Synthetic-series unit tests** (252 passing, incl. 21 in `tests/test_allocate.py` for the allocation engine — risk/sector/capital cap invariants, degenerate cases, method divergence on engineered volatility dispersion, and a regression test for the aggregate-capital-cap bug caught via live testing; 22 in `tests/test_backtest.py` for the screen backtester — engineered-edge-vs-null universes, dedup/entry-convention/baseline/cost arithmetic, bootstrap determinism, the <30-event suppression floor, a JSON-serialization regression test, and vectorizer≡evaluator consistency checks across cheap, expensive-symbol, expensive-cross-sectional, and weekly-timeframe specs; 36 in `tests/test_universes.py` for the universe registry — lookup/error message, path-helper and liquidity-gate defaulting, migration idempotency, per-universe symbol fetch/caching for all three universes (including nse_etf's curated-symbol filtering), and screen-log `universe` field default-on-read; and 13 in `tests/test_evaluator.py` for the webapp selector's list/switch/422 contract, the survivorship-note regression, the sector-data-gap warning (helper-level and live-endpoint), and preset `universes`-tag computation): constructed price paths with known correct answers — an engineered pullback-to-EMA50 must match `support_at_ma` while a breakdown through the same EMA must not; a range-bound series must yield levels at both extremes and match `near_support` while rejecting `near_resistance`; a golden cross is verified at its exact bar; pivots must not exist in the final k bars; relative strength must reject a stock benchmarked against itself; a missing benchmark must raise; a synthetic multi-sector universe with engineered momentum dispersion drives the cross-sectional pre-pass tests, including a dedicated look-ahead test (two sectors whose relative performance flips between an early as-of date and today must rank in opposite order at each date); a v0.9 volatility-dispersion universe (four symbols with identical drift but different intraday range) cleanly separates `atr_pct_percentile` into quartiles, and a skip-month-vs-raw-return universe (a spike confined to the most recent month) proves `mom_12_1`-basis `rs_percentile` ranks differently from `return`-basis on the same data — the entire point of the skip-month construction; each of the 7 new v0.9 presets is checked against an engineered match case and an engineered rejection case, and every preset's `evidence` object is schema-checked (valid `basis`, non-empty `finding`/`caveat`, sources required when `basis="academic"`). **Golden-query suite** (23 fixtures): hand-verified query → expected-DSL pairs, including two mandatory refusals (a P/E query, an unmapped sector name) and 4 new v0.9 fixtures (12-1 momentum, low/high volatility, the full Minervini "stage 2" conjunction). The offline half runs in CI and asserts every fixture still validates as the DSL evolves; the live half (`python -m tests.golden_harness`, needs an API key) scores the actual parser with semantic comparison — defaults filled, conditions sorted — and must be N/N before any parser-prompt change ships. **Live-data sanity checks** (manual, first run): backfill symbol count vs 500; spot-check three symbols' EMA50/RSI against a charting platform; run the flagship query and eyeball charts of the matches. All 7 new v0.9 presets were additionally run against the live 500-symbol store to confirm sane, non-degenerate match counts before shipping.

## 11. Operations runbook

First run: create a Python 3.10+ venv (`python3.12 -m venv .venv && source .venv/bin/activate` — both entry points guard against older interpreters and print this fix), `pip install -r requirements.txt`, set `ANTHROPIC_API_KEY`, `python -m screener.cli backfill` (10–15 min), then `python -m screener.cli verify` — an automated 13-check health report (coverage vs index list, freshness, depth, bar integrity, duplicates, corporate-action smell test via >40% single-day move counts, benchmark presence, RSI/EMA spot checks on a 25-symbol sample, the data-layer-v2 cross-source check §4a, and screen-log JSONL integrity) that exits non-zero on FAIL for pipeline gating; `verify --jumps` lists the exact bars behind the adjustment smell test with split-ratio hints, and `refetch SYMBOL` drops and freshly re-downloads one symbol as the unadjusted-data remedy. Nightly: cron `python -m screener.cli update` after 18:30 IST (add `bhavcopy-update` alongside it once side-by-side collection should run unattended — see §4a). Quarterly: refresh the universe and re-backfill to pick up rebalances. Screens: `screen "<text>"`, `--dry-run` to inspect the compiled spec, `--json` to bypass the LLM, `--as-of YYYY-MM-DD` to replay a historical date (CLI parity with the web UI's date picker), `--out file.csv` to export. Backtests: `backtest "<text>"` (or `--preset id` / `--json`) runs the same spec as an event study — `--horizons`, `--cooldown`, `--cost-pct`, `--stride`, `--hypothesis`, `--no-sensitivity` — a readable text report with the survivorship caveat always printed last. `--universe` is accepted on `backfill`/`update`/`verify`/`screen`/`backtest` (§12g) — `nifty500` (default) or `nse_full`; the webapp has an equivalent header dropdown. Failure modes: stale-store error → run update; universe fetch fails → cached list used automatically, manual CSV drop-in as last resort; yfinance thin/missing data for a symbol → excluded by the 60-bar minimum, check the alias problem in §4; parser refusal → the query needs rephrasing within scope, or the concept belongs on the roadmap.


## 12. Web UI and evidence layer

`python -m screener.webapp` serves a single-page interface (FastAPI backend, dependency-free vanilla-JS frontend in `web/index.html`, port 8501) built around full auditability of every screen. The page always shows, together: the original query, the plain-English compiled interpretation, the raw JSON spec (collapsible), the data as-of date and mode, and funnel statistics (universe → liquidity-excluded → evaluated → matched).

**Screen definition** offers three routes: plain-English (LLM parse with mandatory interpretation echo), raw JSON spec, and a grouped **preset dropdown** backed by `presets.py` (§9a) via `GET /api/presets`, which shows each preset's rationale and compiled English before running. An **as-of date picker** replays any historical date; when set, condition evaluation, match metrics, and sparkline windows are all computed at that row (never the latest bar), preserving the reproducibility guarantee for historical screens. Note the survivorship caveat from §4 applies to any historical replay.

The core of the UI is the **evidence trail**, powered by `explain.py`. For every condition on every displayed stock it reports pass/fail plus the observed values behind the decision — e.g. for `support_at_ma`: the date the low touched the MA and the touch distance, the worst close-vs-MA excursion across the window, and the latest close's margin above the MA. Pass/fail is delegated to the same `evaluator.cond_*` functions used by the screen itself, so the explanation can never disagree with the result; `explain.py` only adds observability. Each match card also renders an **evidence sparkline**: an inline SVG of the last 60 bars (up to the as-of row) overlaying exactly the series the spec referenced (EMAs, bands, 52-week levels) and every horizontal level the evidence produced (swing support/resistance, breakout levels) — so verifying a touch or a level does not require leaving the page. A **near-miss** section lists stocks failing exactly one condition with the failing condition marked ✗ — the most useful feedback for tuning tolerances; a toggle button hides/shows it without losing the funnel-stat count. Match cards are capped at `MAX_MATCHES` (100) in the payload — `stats.matched` always reports the true total, so a loose filter's size is never hidden, just not rendered as hundreds of DOM cards; a note above the results explains the cap when it's active. A client-side **CSV export** button downloads the currently-displayed matches (symbol, name, industry, and every snapshot metric) without a dedicated export endpoint, since the data is already in the page.

Every run is appended to `data/screen_log.jsonl` — timestamp, as-of date, full spec, its plain-English description, funnel stats, matched symbols — turning the deterministic-replay guarantee into a browsable history (`GET /api/log`, or `python -m screener.cli log`). Log writes never raise; a failed write cannot break a screen. A **recent screens** panel in the UI fetches this log and lets you replay any past run with one click (restores the spec into the JSON tab and the as-of date), refetching whenever a new screen completes so it never shows stale entries.

Endpoints: `GET /api/status`, `GET /api/health` (§12a), `GET /api/presets`, `GET /api/log`, `POST /api/parse` (text → spec via the LLM; structured error without an API key), `POST /api/screen` (spec → matches + near-misses + sparklines + methodology block). When `data/prices.parquet` is absent the backend boots a labelled synthetic 11-stock demo universe (`demo.py`) with engineered behaviours — including three exemplars whose only purpose is exercising the data-quality flags in §12a — so the UI, API, and evidence layer are fully testable — in CI and by a fresh clone — without market data.

## 12a. Operational robustness (v0.7 Item 6)

**Stale-server fix (P0).** `webapp._load_state()` used to load panels once at process startup and cache them forever — a long-running server would silently keep screening yesterday's data after the nightly `python -m screener.cli update` wrote a fresh `prices.parquet`. It now compares the store's mtime on every call and rebuilds (clearing the cross-sectional pre-pass cache too, since it's keyed by `id(panels)` and a GC-reused id could otherwise serve stale ranks) whenever the file has changed underneath it. Verified with a test that swaps the underlying parquet file mid-session and confirms `as_of` updates without a restart — not just that the code compiles.

**Data-quality flags.** Every match/near-miss now carries a `flags` list computed from signals `verify`'s checks already look for, but scoped to one symbol and surfaced where a user is actually looking: `jump` (a >40% single-day move within the spark window — possible unadjusted split/demerger, "levels may straddle a gap"), `thin_history` (fewer than 250 bars as of the as-of row — long-lookback indicators unreliable), `stale` (the symbol's last bar predates the store's overall latest — possibly suspended/delisted). The UI renders a small `⚠ N data flag(s)` badge with the reasons in a tooltip. `demo.py`'s JUMPY/THINHIST/STALECO exist solely to exercise these; live-tested against the real store too — it correctly flagged a genuine recently-listed Nifty 500 name as `thin_history`.

**Config overrides.** `config.py` loads an optional `data/config_local.toml` at import time, overriding any of a fixed allowlist (`_OVERRIDABLE`): the liquidity gate, staleness window, the four SR pivot/clustering constants (moved from `sr.py` into `config.py` so overrides actually take effect — `sr.py` keeps them as module aliases for compatibility), and the web UI's spark-bar count / match cap. Unknown keys in the file are flagged and ignored, never silently creating an unused setting. `config.config_hash()` — a short hash of every overridable tunable's effective value — is logged with each screen-log entry and shown in the methodology footer, so a screen's result is reproducible together with the config that produced it, not the spec alone. Requires `tomli` on Python <3.11 (stdlib `tomllib` on 3.11+).

**Parser resilience.** `parser.parse_with_assumptions()` retries once, with a corrective instruction, if the LLM's response isn't valid JSON — a prompt-following failure, not a scope refusal. If both attempts fail, or the resulting spec fails `dsl.validate()`, the query and raw output are appended to `data/parse_failures.jsonl` as a vocabulary-improvement backlog; legitimate `{"error": ...}` scope refusals are deliberately *not* logged there, since that's the parser working as intended. The LLM may optionally return a top-level `"assumptions"` list naming any canonical default it filled in because the user gave no explicit number — stripped from the spec before validation, surfaced by `/api/parse` and rendered in the UI as "interpreted with defaults: …". `parser.parse()` remains a thin backward-compatible wrapper returning just the spec.

**`/api/health`.** Cheap JSON for cron/uptime monitoring, cheap enough to curl after every nightly `update`: mode, as-of, store mtime (confirms the P0 fix actually picked up the fresh file), panel count, benchmark presence, log writability, `git describe --always --dirty` as the deployed version, and the config hash.

**Screen-log rotation.** Once `data/screen_log.jsonl` exceeds 5,000 lines, the oldest entries move into `data/screen_log.rotated.jsonl` rather than growing the active file forever or discarding history. `verify.check_screen_log()` validates both files' JSONL integrity together.

**Golden harness in CI.** `.github/workflows/golden-harness.yml` — `workflow_dispatch` only (needs an `ANTHROPIC_API_KEY` repo secret to hit the real API), so a parser-prompt PR can be gated on 19/19 without requiring a local run. The existing `ci.yml` push/PR job remains offline-only.

## 12b. UI depth (v0.7 Item 5) — daily-driver features

**Screen diff ("what changed since last run").** `dsl.spec_hash()` — moved here from what used to be golden_harness-only canonicalisation logic (`dsl.canonicalize_conditions`, now shared by both) — hashes a screen's logic + conditions (defaults filled, sorted, **as_of excluded**: rerunning the same criteria on newer data is still "the same screen"). Every screen-log entry now carries its `spec_hash`; `/api/screen` looks up the most recent prior entry with a matching hash, diffs the matched-symbol sets, and re-explains each dropped symbol against *current* data so the UI shows exactly which condition now fails rather than just that a symbol vanished. Surfaced as a "+N new / −M dropped" badge with an expandable dropped list; new matches get a small NEW tag on their card.

**Full chart modal.** `_referenced_fields`/`_evidence_levels` were factored out of the existing inline-spark builder so a new `_chart_payload` (CHART_BARS=250, full OHLCV) could reuse them without duplicating the overlay logic. `POST /api/chart` fetches one symbol's full window lazily on click — embedding 250 bars per match in every `/api/screen` response would have bloated the main payload for no benefit, since most matches are never opened. The frontend renders hand-rolled SVG candlesticks (no external charting library — the offline-first constraint from §1 stands) with a volume subpane and drag-to-zoom (mouse down/up over the SVG re-renders a narrower `[start,end]` bar range client-side, no server round-trip).

**Watchlist with signal-decay tracking.** `POST /api/watchlist` tags a match: symbol, the as-of date and close *at the moment of tagging*, and the full spec (so what was being checked is part of the record, not just the symbol). `GET /api/watchlist` re-evaluates that same spec against **today's** data for every entry — current close, % move since tag, and `still_holds` (the original conditions, checked now, not just once) — turning a star into a decay tracker rather than a static bookmark. `DELETE /api/watchlist/{id}` rewrites the file without that line (small enough collection that read-modify-write is fine; no need for the log's append-only/rotation treatment).

**Saved custom screens.** `data/user_presets.json`, a mutable list (not JSONL — unlike the screen log, entries get renamed and deleted). Validated identically to the built-in library at save time (`dsl.validate()`, same as `presets.py`'s own import-time check) so a bad spec is rejected on save, never discovered later on run. The frontend merges these into the same `PRESETS` array the built-in dropdown already uses (`id` prefixed `user:` to avoid collisions), so the existing preset-selection code path needed no changes — just a "My screens" optgroup and a small manage-panel for rename/delete.

**Multi-screen dashboard.** `_run_screen(spec)` is the extraction that made this cheap: it's the entire body of what `/api/screen` used to inline, now callable per-preset. `POST /api/screen_batch` runs N built-in-or-saved presets through it and returns just (matched count, top-3 symbols by 3-month return, new-since-last-run) per screen — the existing diff/log machinery just works, since each batch entry is a real screen run in its own right. The UI's checkbox picker builds its list from the same merged `PRESETS` array as the main dropdown.

**Results table ergonomics.** The per-match card renderer (`renderCard`) moved to module scope so a dedicated `renderMatchesSection()` could re-render just the matches list — sorted by a `SORT_CMP` table (return/RSI/price/symbol, asc or desc) and filtered to one industry via sector chips built from the current result set — without a server round-trip. The "Matches (N)…" header is `position: sticky` so it stays visible while scrolling a 100-card list; near-misses are intentionally left unsorted/unfiltered, since they're a secondary, usually-shorter list.

All six were driven live against the real server with a temporary Playwright harness (screenshots + console-error checks), the same discipline as v0.6.2 and the Item 6 robustness work — not just unit-tested.

## 12c. Evidence-based strategy presets (v0.9 Item 9)

**`LITERATURE.md` came first.** Per the roadmap's explicit sequencing, the review document was written before any preset or indicator it justifies — a preset that can't point to a section there doesn't ship as evidence-based. It covers eight price/volume anomaly families (cross-sectional momentum, 52-week-high anchoring, time-series momentum, moving-average rules, volume-confirmed momentum, the low-volatility anomaly, the practitioner Minervini/O'Neil "stage 2" template, and consolidation breakouts), each with the canonical papers, the core finding, India-specific evidence where it exists, and an honest caveat — including, deliberately, the ones that undercut the family's own tradability (momentum crashes, the Sullivan-Timmermann-White data-snooping critique of moving-average rules, survivorship bias in practitioner track records).

**Preset schema extension.** Every preset in `presets.py` (26 total, up from 19) now carries an `evidence` object: `{"basis": "academic"|"practitioner"|"mixed", "sources": [...], "finding": str, "caveat": str}`. `"academic"` requires at least one source (schema-checked in tests); presets with no dedicated academic study say so plainly (`"sources": []`) rather than inventing a citation. `GET /api/presets` surfaces the object; the web UI's preset-description panel shows a small colored basis tag plus the finding and caveat *before* the user runs the screen — the same "confirm the interpretation before trusting it" principle the DSL echo already applies to compiled specs, extended to the evidence behind the screen itself.

**Indicator additions** the new presets' DSL mappings needed (§5): `roc_126`/`roc_252` (6/12-month return), `mom_12_1` (the Jegadeesh-Titman 1993 skip-month construction — return from 252 bars ago to 21 bars ago, so a spike confined to the most recent month cannot masquerade as 12-month momentum), `sma_150` and slope fields for every SMA period (needed for the Minervini template's stacked-moving-average check), and the cross-sectional `atr_pct_percentile` condition (§6a) for the low/high-volatility bucket presets.

**Seven new presets**, each pointing at a specific LITERATURE.md section: `momentum_12_1_leaders` (§1, academic), `near_52w_high_ghw` (§2, academic), `tsmom_regime` (§3, academic), `ma_timing_highvol` (§4, mixed — MA rules restricted to the high-volatility tercile where Han-Yang-Zhou found the effect concentrated), `volume_momentum` (§5, academic), `lowvol_defensive` (§6, academic), and `minervini_stage2` (§7, explicitly practitioner — labeled as such, not dressed up). All 19 pre-existing presets were annotated retroactively rather than left unlabeled; several map only loosely to the eight vetted families (candlestick patterns, sector-rank combinations) and are honestly marked `"practitioner"`/`"mixed"` with a finding stating no dedicated study was reviewed, rather than a fabricated citation.

## 12d. Portfolio allocation engine (v0.10 Item 10)

`screener/allocate.py` turns a ranked list of already-screened symbols + capital + a risk tolerance into integer-share position sizes. It is explicitly framed — in this module's docstring, the API response (`disclaimer` field), and the UI — as a **sizing calculator with documented methodology, not a recommendation engine**: it has no view on which stocks to buy, only on how much of each. Non-goals, deliberate: no mean-variance optimisation (estimation error dominates on small screened subsets — DeMiguel, Garlappi & Uppal 2009), no Kelly criterion (drawdown profile unsuitable for discretionary use), no return forecasts, no auto-execution.

**Three methods**, always computed against the same caller-ranked symbol list (this module never re-ranks — the caller, e.g. the UI's current results sort, decides priority when candidates exceed the caps):
- **`risk`** (default) — fixed-fractional risk sizing (Van Tharp/Turtle-style): per-position risk = capital × risk_pct%, stop distance = 2×ATR(14), shares = floor(risk ÷ stop distance).
- **`inverse_vol`** — naive risk parity: capital weight ∝ 1/ATR% (calmer names get more capital).
- **`equal`** — 1/N. Computed and returned as an always-present `baseline` key alongside whichever method the caller actually requested (unless they requested `equal` itself) — DeMiguel et al found 1/N is the benchmark no optimiser reliably beats out-of-sample on estimated inputs, so it's shown for comparison rather than buried as just another dropdown choice.

**Constraints applied to every method, in this order**: per-position cap (`max_position_pct`, default 15% of capital), sector cap (`sector_cap_pct`, default 30% of capital per industry, enforced by tracking running per-sector deployed value and trimming/excluding later same-sector candidates), and — the aggregate capital cap — a running `capital_deployed` total that trims or excludes later candidates once the sum would exceed the input capital. **This last cap was missing in the first implementation and caught only via live testing against the real 500-symbol store**: nine candidates each individually satisfying the 15% position cap and passing distinct sector caps summed to ₹104,193 against a ₹100,000 input, because `risk` sizes each position independently off the risk budget with no built-in awareness of the running total. Fixed by tracking `capital_deployed` the same way `sector_deployed` already was, with a dedicated regression test (`TestAggregateCapitalCap`, 9 same-drift/distinct-sector symbols) — synthetic tests with only 4 symbols hadn't summed past capital by chance, which is why this shipped past unit tests and was only caught live.

Positions below `min_ticket` (default ₹5,000) or with NaN/non-positive ATR (insufficient history — never sized blind, the same discipline as the rest of the codebase) are excluded with a stated reason, never silently dropped or zero-filled. Each position carries a plain-English `rationale` string in the same evidence-ledger style as screen conditions (e.g. `"risk ₹1,000 ÷ stop distance ₹59.65 = 13 shares = ₹14,424 (14.4%)"`).

**API**: `POST /api/allocate` — `{symbols, capital, method, risk_pct, max_positions, max_position_pct, sector_cap_pct, min_ticket, as_of, spec}` → `{method, capital, positions, excluded, summary, baseline?, disclaimer}`. Logged to `data/allocation_log.jsonl` — a sibling to `screen_log.jsonl` kept as its own file rather than interleaved, since the schemas genuinely differ (a position table, not a matched-symbol list) and `verify.check_screen_log`'s required-keys check would otherwise need to special-case allocation entries; if the caller passes the originating screen's `spec`, its `spec_hash` is logged too, tracing an allocation back to the screen that produced its symbol list.

**UI**: an "💰 allocate" button on the results panel opens a form (capital, a conservative/moderate/aggressive/custom risk-preset selector, method toggle, constraint fields) that sizes the current match list in its existing sort order; results render as a position table plus (for `risk`/`inverse_vol`) the equal-weight baseline table underneath for comparison, an excluded-with-reasons list, the disclaimer, and a CSV export. Resets when a new screen runs, the same discipline as the recent-screens panel.

## 12f. Screen backtester (v0.12 Item 14) — event-study engine

`screener/backtest.py` answers: for any DSL spec, what happened after this signal fired historically, versus just holding the universe on the same dates? It is explicitly an **edge detector for filters, not a portfolio simulator** — no position sizing, no compounding, no stops, no execution modelling (that's `allocate.py`'s job, applied once a screen has earned some trust here).

**Locked methodology.** Event = signal True at bar t AND no event for that symbol in the prior `cooldown` bars (default 20) — de-dupes "signal stays true for a week" into one event, not near-duplicate samples. Entry at **open[t+1]** (bar-t close is where the signal fires; a close-t entry would be look-ahead); forward return at horizon h = close[t+h] / open[t+1] − 1. An event within h bars of its panel's end is excluded from *that horizon only*, never NaN-polluted into the stats for the others. Baseline is the same-date equal-weight mean forward return across all liquidity-passing universe symbols — the part most home-built backtests get wrong, since it controls for the market regime on the signal date rather than comparing to an unconditional long-run average. Costs: gross and net (net = gross − a configurable round-trip %, default 0.30) always shown side by side. Statistics: per-horizon count/mean/median/hit-rate/p5/p95/worst-5%-mean on the excess distribution, a date-level block bootstrap (seeded, reproducible) on the event-date portfolio series — never pooled-event t-stats, since same-date events across symbols are cross-correlated — and an outright "insufficient events" suppression below 30 valid events at a horizon. An optional free-text `hypothesis` is logged with every run (multiple-testing is the real killer; writing the expectation down before seeing the answer is the cheapest available discipline). A survivorship caveat string is attached to every result, unconditionally — the current-constituent universe flatters dip-buying/mean-reversion setups, since names that didn't survive to today aren't in it to drag the average down.

**Vectorized signal path.** Per-condition vectorizers (`_vec_*` functions in backtest.py) reproduce every cheap evaluator condition (compare/range/change/trend/cross/proximity/support_at_ma/volume_spike/gap/tight_range/flat_base/candle/rel_strength/sector, plus weekly-timeframe variants via an as-of expansion) as whole-panel pandas boolean Series, built to match `evaluator.py`'s per-bar semantics exactly (including edge cases like `support_at_ma`'s NaN-in-window override). The expensive set — `near_support`, `near_resistance`, `breakout_resistance`, `bb_squeeze` (all O(lookback) pivot/percentile work per call) and the cross-sectional `rs_percentile`/`sector_rank`/`atr_pct_percentile` (each needing a universe-wide rank rebuild) — instead sample the *existing* per-bar evaluator functions on a `stride`-bar date grid and forward-fill between samples, an explicit, documented approximation rather than a silent one.

**Acceptance test, not a runtime guarantee.** `verify_vectorizer_consistency()` samples random (symbol, date) pairs and asserts the vectorized signal matches `evaluate_symbol()` exactly. For cheap-only specs this holds at every date, so the test samples freely. For specs touching an expensive type, exactness is only guaranteed *at the stride-grid dates themselves* (the cross-sectional path literally reuses `cross_section.build_cross_section()`'s own cache at those dates, so it's the same computation twice by construction) — so the test restricts its sample to grid-aligned dates for those specs, testing the guarantee actually made rather than a stronger one nobody promised.

**Perf reality vs. the original plan.** The spec's example stride (5 bars) was measured against the real 500-symbol/5-year store before shipping: `near_support`/`breakout_resistance`/`bb_squeeze` at stride=5 took roughly 6-15 minutes for a *single* condition — well past any reasonable "<3min" budget. `stride=20` (~1 trading month between samples) measured at ~25-90s per expensive condition type and ~30-40s for a cross-sectional one, comfortably inside budget, and became the shipped default rather than the originally-planned 5 — documented here rather than silently kept at a number that didn't hold up. A cheap-conditions-only spec runs in ~13-19s regardless of stride. The auto-detected one-at-a-time **sensitivity grid** (perturbs each numeric threshold ±2 steps, verdict "robust across range" vs. "edge concentrated at one value — treat as curve-fit") fully reruns the signal path per variant, so it costs roughly (1 + 4×n_params)× the core runtime — for a spec with one sensitivity-eligible parameter and an expensive condition type, that can mean several minutes; it's exposed as an independent UI/API/CLI toggle (`sensitivity: bool`) for exactly that reason, not forced on.

**Cross-sectional history, simplified from the original plan.** Rather than a new vectorized date-indexed rank rebuild in `cross_section.py`, the shipped implementation calls the *existing* single-date `build_cross_section()` once per stride-grid date (not once per bar, and not a new code path) and forward-fills between samples the same way the per-symbol expensive types do. Measured fast enough in practice that the heavier rewrite wasn't needed; worth revisiting only if a future spec combines multiple distinct cross-sectional windows and this becomes the bottleneck.

**Two bugs found via live testing, fixed before shipping.** (1) The very first real `/api/backtest` call 500'd: `json.dumps()` rejects a raw Python `nan`, which a horizon-truncated event's row legitimately contains — fixed with an explicit NaN→`None` pass over the events table (`_events_to_records`) before it leaves `backtest_spec`. (2) This item's own test suite — which builds and discards many short-lived synthetic `panels` dicts — started intermittently corrupting *unrelated* preset tests elsewhere in the same pytest run. Root cause: `cross_section.py`'s in-process cache keyed on `id(panels)` alone, a risk its own docstring already named ("a GC-reused id could otherwise serve stale ranks indefinitely") but that no prior test suite had exercised heavily enough to trigger. Fixed by adding `frozenset(panels)` (the symbol set) to the cache key, with a regression test that plants a stale entry under a real id() with a different symbol set and confirms the genuine computation wins rather than the stale hit.

**API + UI + CLI.** `POST /api/backtest` — `{spec, horizons, cooldown, cost_pct, min_turnover_cr, stride, min_events, hypothesis, sensitivity}` → per-horizon stats, the per-event table, an event timeline, the sensitivity grid, and the survivorship note; logged to a new `data/backtest_log.jsonl` (spec hash + params + hypothesis + per-horizon summary — not the full per-event table, which alone can run to thousands of rows). The UI's "🧪 backtest" panel (opened from the results view, same pattern as "💰 allocate") replays the current spec: a form for all the run parameters, per-horizon tables (gross/net excess, hit rate, p5/p95, worst-5%, the bootstrap CI), an excess-return histogram vs. baseline, an events-per-month bar chart ("a signal that stopped firing is a finding"), the sensitivity table with its verdict line, the survivorship banner, and a CSV export of the per-event table. `python -m screener.cli backtest "<query>" [--preset id] [--json] [--horizons ...] [--cooldown N] [--cost-pct X] [--stride N] [--hypothesis "..."] [--no-sensitivity]` gives the same engine a readable text report. All three verified live against the real 500-symbol store; the UI additionally verified via Playwright (zero console errors, correct rendering with the sensitivity grid on and off, print-mode visibility).

This is a distinct, sizeable follow-up (running and curating 26 backtests, settling on consistent robust/fragile wording) rather than a natural extension of shipping the engine — revisit as its own scoped effort.

## 12h. Cohort tracker (v0.14 Item 16) — walk-forward out-of-sample filter validation

`screener/cohorts.py` answers the question the backtester structurally cannot: does this filter's edge hold up on names nobody hand-picked, followed forward in real time, with the ones that go bad NOT quietly excluded? It is the out-of-sample complement to §12f's in-sample event study — a cohort freezes a set of matches (or a sized allocation) at signal time and tracks them forward at the *exact same* horizons/baseline/entry convention as the backtester, so IS and OOS numbers for the same spec are directly comparable. Like the backtester, this is a **filter validator, not a trade tracker**: no fills, no partial exits, no P&L accounting.

**Dates frozen, never prices.** A cohort record stores `entry_date` (a string) and nothing price-derived until a milestone freezes; returns are recomputed from the live store on every read until frozen. This makes cohort returns automatically invariant to a retroactive split/bonus adjustment rewriting the store's history — percentage returns are scale-invariant by construction, verified by a dedicated test that halves an entire synthetic price series and asserts the recomputed return is unchanged, not just asserted by inspection.

**Entry convention, reverse-engineered to match the backtester exactly.** `entry_date` is the first trading day after the cohort was created (`_next_trading_day`, using the union of dates across the cohort's own symbols). The day *before* `entry_date` plays the backtester's signal-bar `t`; `_signal_row()` finds that row by searching a symbol's own panel for `entry_date` and stepping back one, so `backtest.py`'s h-bar-counting convention (`close[t+h] / open[t+1] - 1`) applies unchanged. Day-0 is `pending`: no returns, ever, until the entry bar actually exists in the store — a cohort created "today" stays pending until `refresh_cohort()` runs again after tomorrow's bar lands, exactly mirroring how a real signal can't be acted on until the next session opens.

**Milestone freezing vs. the live "current" view.** At 5/20/60 bars post-entry (`HORIZONS`, counted the same way the backtester counts h), per-symbol and cohort-aggregate returns compute once and freeze permanently into the record (`_aggregate_milestone`) — gross, net (at the universe's default cost), the baseline value looked up for that exact signal date, and excess (gross and net). A live, never-frozen "current" mark-to-market view (`current_snapshot`) is separately available for the active-cohorts list, using each symbol's latest close as an undated exit; it is recomputed on every read and never persisted, so it keeps drifting for as long as a cohort has an entry date. At 60 bars, status moves to `completed` — still fully present in scorecards, never archived out.

**Baseline parity — a shared function, not a second implementation.** `backtest.compute_baseline(panels, symbols, horizons, min_turnover_cr)` was factored out of `backtest_spec()`'s own inline baseline computation specifically for this item: same-date equal-weight mean forward return across all liquidity-passing universe symbols, indexed by date, per horizon. Both the backtester and the cohort tracker call this one function — a dedicated "baseline parity" test asserts the cohort's baseline lookup for an engineered date matches `backtest_spec()`'s own baseline for the same synthetic universe exactly, closing off the risk of the two engines silently drifting apart on their most important shared number. (`backtest.py` also gained two other renames as part of this refactor, `_fwd_return_series`→`fwd_return_series` and `_liquidity_series`→`liquidity_series` and `_common_dates`→`common_dates`, all made public because the cohort tracker needed them directly — no behavior change, all call sites updated.)

**Symbol lifecycle — the whole point of the tool.** `_milestone_reached()` checks "has horizon h elapsed" against the *full universe's shared date calendar* (`backtest.common_dates(panels)`), not any one symbol's own panel — this is what lets `_symbol_return()` correctly distinguish "this symbol delisted/got suspended early" (its own panel doesn't reach that far even though the cohort as a whole has aged enough — flagged `stale`, last available close carries forward as the exit price) from "the cohort just hasn't aged that far yet" (returns `None`, not a stale flag). A stale symbol is never dropped from the aggregate; the backtester's survivorship bias comes from exactly this kind of quiet exclusion, and avoiding it is this tool's entire reason for existing over just re-running the backtester more often.

**Weight normalization.** `weights_from_symbols()` gives equal weight ("track these matches"); `weights_from_positions()` derives weight from each position's share of the *tracked* capital — the sum of the positions themselves, not the original allocation's input capital, so undeployed cash doesn't dilute the measured return. Both paths always sum to 1.0, so equal- and allocation-weighted cohorts aggregate through the identical weighted-mean code path.

**Scorecard — small-N honesty.** `scorecard(universe_id, spec_hash, ...)` aggregates every cohort sharing a spec_hash into per-horizon OOS stats (mean/median/hit-rate on excess net), and looks up the most recent logged backtest for the same spec_hash as its IS comparison — filtered on **both** `spec_hash` *and* `universe`, not spec_hash alone, since the same DSL spec can legitimately run against more than one registered universe and pairing an nse_full cohort with a nifty500 backtest would silently compare apples to oranges. (This surfaced a real gap: the backtest log had no `universe` field before this item — added to `_log_backtest()`'s logged entry and to the CLI's own `backtest` command, with a regression test confirming the field is populated for the default universe.) Below 20 total tracked names at a horizon, the mean is suppressed as "insufficient sample" rather than printed with false confidence — the same evidence-accumulation-not-hypothesis-testing posture the backtester's own <30-event floor takes (§12f).

**Storage — full-rewrite JSONL, not append-only.** `data/{universe}/cohorts.jsonl` is read-all/mutate/write-all on every change (`_load_all`/`_save_all`), unlike the codebase's other append-only logs (screen/allocation/backtest), because a cohort record has a genuine mutable lifecycle (pending → active → completed, with milestones filling in over calendar time) rather than being a one-shot event.

**Lazy refresh — "viewing is refreshing," no cron state.** There is no scheduled job for cohort milestones. `list_cohorts()`/`get_cohort()` call `refresh_cohort()` on every read, which resolves a pending cohort's entry date, freezes any newly-reached milestone, and persists the change if anything moved — exactly the ROADMAP spec's explicit "no cron state" requirement. `create_cohort()` accepts an optional `created_ts` override purely for tests, letting a synthetic cohort be backdated 60+ trading days without waiting two real months for milestones to become reachable.

**Two real bugs found while building this, fixed before shipping.** (1) `tests/test_evaluator.py` posts to `/api/screen` in roughly 17 places across many test classes; `_run_screen()` unconditionally logs every screen to `webapp.LOG_FILE`, and most of those call sites had no isolation fixture — every test-suite run was silently appending demo-symbol entries to the real `data/screen_log.jsonl` (over a thousand contaminating lines had accumulated there before this was caught). Fixed with a new `_isolate_webapp_log_files` autouse fixture in `tests/conftest.py` that monkeypatches all six of `webapp.py`'s module-level log/store path constants (`LOG_FILE`, `ROTATED_LOG_FILE`, `ALLOCATION_LOG_FILE`, `BACKTEST_LOG_FILE`, `WATCHLIST_FILE`, `USER_PRESETS_FILE` — all bound once at import time from `config.DATA_DIR`, which is *why* a test-local `monkeypatch.setattr(config, "DATA_DIR", tmp_path)` alone was never enough to redirect them) to a per-test `tmp_path`, closing every exposure point at once rather than patching call sites one at a time; verified byte-identical `data/screen_log.jsonl` before and after a full suite run. (2) `cli.py`'s own `screen` command never wrote to `LOG_FILE` at all (only the webapp route did) — meaning `cohort create --from-last-screen`, built as part of this same item, was silently broken for CLI-only workflows despite its own error message promising it would work after `screen`. Fixed by having `cmd_screen` call the same `webapp._log_run()` webapp screens already use, so CLI and web-UI screens share one history and either surface can seed a cohort from the other's run.

**API + UI + CLI.** `POST /api/cohorts` (`{spec, symbols|positions, method, notes}` — exactly one of `symbols`/`positions`), `GET /api/cohorts[?spec_hash]`, `GET /api/cohorts/{id}` (each response gets a fresh `current` field attached, never stored), `GET /api/scorecard/{spec_hash}`. ## 12i. Cohort replay & performance engine (v0.15 Item 17) — in-sample cohort testing

`screener/cohort_perf.py` extends §12h's cohort tracker along two axes: a cohort can now be created as of any historical date instead of only "starting now" (**replay** mode), and every cohort — replay or forward — gets a full performance panel over an arbitrary window, not just the fixed 5/20/60-bar milestones. Both are additions to the Item-16 record and its baseline/entry-convention machinery, not a fork: `cohorts.py` gained a `mode`/`as_of` field pair and a scorecard wall; `cohort_perf.py` is new, pure, and reuses `cohorts._signal_row`, `backtest.common_dates`, and a new sibling of `compute_baseline`.

**The integrity wall.** A cohort's `mode` (`"forward"` or `"replay"`) is derived once, at creation, from whether an `as_of` was given — never a caller-settable field (`create_cohort()` has no `mode` parameter at all, so there is nothing to override even maliciously; a dedicated test asserts this by inspecting the function's own signature). A replay cohort is in-sample by construction: the date was chosen with everything that happened after it already visible, exactly the bias the backtester carries and the forward tracker exists to avoid — so replay cohorts are walled out of `cohorts.scorecard()`'s OOS aggregate (`_aggregate_scorecard_horizons()`, factored out of the old inline loop specifically so the same aggregation code serves both the OOS `horizons` block and a separate, clearly-labelled `replay` block, rather than a second reimplementation that could silently drift). `REPLAY_SURVIVORSHIP_NOTE` is attached to a replay cohort's API response the same way `current` is (`webapp._attach_current`, a response-shaping addition, not a stored field) — this was itself a gap caught on a post-implementation re-read of the locked spec: the note existed as a constant but was never actually wired into any surface until a dedicated test (`test_replay_cohort_carries_survivorship_note_on_read`) was written to prove it reached the API.

**Replay entry resolution — one function, two anchors.** `refresh_cohort()`'s pending→active step picks its anchor date based on `mode`: a forward cohort anchors to `created_ts` (unchanged from Item 16 — "now," so it waits for a real future bar); a replay cohort anchors to its own `as_of` instead. Since `as_of` is already historical and already validated to have a later bar, a replay cohort resolves to `active` (often straight through to `completed`) on the very next `refresh_cohort()` call — no waiting on real calendar time, which is the entire value of replay. `as_of` itself is canonicalized by `_resolve_as_of()`: resolved leniently to the trading day at-or-before the given date (the same semantics `evaluator._row_at` already uses for every other as-of picker in the app — a weekend/holiday date rolls back rather than erroring) and validated to leave at least one later bar in the shared universe calendar, raising `ValueError` rather than silently creating an unevaluable cohort.

**The performance engine (`screener/cohort_perf.py`).** `evaluate_performance(cohort, panels, universe_symbols, min_turnover_cr, end_date=None, benchmark=None, cost_pct=...)` is a pure function serving both modes identically — a forward cohort defaults its window to entry_date→latest bar; a replay cohort gets the same computation with an already-long window available instantly. `_resolve_window()` clamps `end_date` to the latest bar and resolves non-trading days to the prior one, against the *full* universe calendar (`backtest.common_dates(panels)`, not just the cohort's own symbols) — the same reasoning Item 16's `_milestone_reached` already established: the shared calendar is what lets one stale symbol be distinguished from "the window just hasn't reached that far yet." `end < entry+1` (no bar strictly after entry) returns `None` — pending semantics, not a zero.

**Equity curves, indexed to 100 at entry, with one documented asymmetry.** The cohort's own curve uses `open[entry_date]` as its 100-point (matching the entry convention exactly — the same price milestone returns already use); the baseline and Nifty curves are indexed to 100 at entry_date's *close* instead, since neither has a single "entry price" the way a traded position does. This means entry_date's own bar reads as a small immediate move for the cohort curve but exactly 100 for baseline/Nifty — a minor, intentional, documented day-0 asymmetry, not a bug (verified directly: `daily_baseline_returns()`'s entry-date contribution is explicitly zeroed before cumulating, so `baseline_curve[entry_date] == 100` always). `backtest.daily_baseline_returns()` is a new sibling of `compute_baseline()`, not a fork of it — `compute_baseline` answers "what did the universe do over the next h bars from a signal date" (Item 14/16's question); an equity curve needs "what did the universe do on this one calendar day," at every date in an arbitrary-length window, which is a genuinely different question the h-bar-forward function structurally cannot answer. Both share the same `liquidity_series()` filtering, so "the universe" means the same thing in both.

**The locked metric set**, all computed once, from the same set of resolved per-symbol windows: cumulative return gross/net (round-trip haircut), excess vs. baseline and vs. Nifty (gross/net), annualised volatility (`daily_rets.std() * sqrt(252)`), max drawdown with peak/trough *dates* (`_max_drawdown()`, cummax/drawdown/idxmin — computed once for the cohort curve and once per symbol, both using the identical helper), hit rates (% positive, % beating baseline over the window), and weighted contributors sorted best-to-worst. Contribution figures use the *renormalized* weight (`weight / weight_total`, not the raw stored weight) specifically so per-symbol contributions sum exactly to the cohort's own gross return even when some symbols didn't resolve — a correctness detail caught by writing a test that checks the sum, not just individual values. **Sharpe is reported only when the window is ≥60 bars** (`MIN_BARS_FOR_SHARPE`); shorter windows return `None` with an explicit `sharpe_note` ("window too short (<60 bars)") rather than an annualised number nobody should trust on a fortnight of data — same posture as the backtester's own <30-event suppression floor and the cohort scorecard's <20-name floor.

**Stale symbols** use the identical carry-last-close-forward rule Item 16 established: a symbol whose own panel ends before the window's end is flagged `stale` in its per-symbol row, never dropped from the weighted aggregate — verified directly (not just re-asserted) via a test that truncates one symbol's panel mid-window and confirms it still contributes.

**A real bug, caught by live UI testing, not unit tests.** The cohort-detail view's initial performance fetch (default window) and a later "evaluate to a different end date" fetch can run concurrently — click "apply" quickly enough after the panel opens, and both requests are in flight at once. The unit tests, which call `evaluate_performance()` directly and synchronously, had no way to expose this; it only surfaced when a Playwright script drove the actual browser end-to-end and the displayed window silently reverted to the default 93-bar result after applying a 9-bar one. Root cause: `loadCohortDetailPerf()` unconditionally wrote its result to the shared `COHORTS_PERF_DATA` global regardless of whether a *newer* request had already landed — whichever fetch happened to finish last won, even if it was answering a now-stale question. Fixed by capturing the requested end-date at the start of each fetch and only applying the response if `COHORTS_VIEW`/`COHORTS_DETAIL_ID`/`COHORTS_PERF_END` all still match what was requested when the fetch began — a stale in-flight response is silently discarded rather than applied.

**API + UI + CLI.** `POST /api/cohorts` gained an optional `as_of` field (deliberately the *only* new field — there is no `mode` field on the request model, which is itself the enforcement mechanism for "not user-overridable," not a separate validation check). `GET /api/cohorts/{id}/performance?end=...` returns the full metric set; 404 for an unknown cohort, 422 if the window isn't evaluable yet. The UI's "track these matches"/"track this portfolio" buttons now read `LAST_SPEC.as_of` and thread it through automatically — running a screen with an explicit as-of date and then tracking it produces a replay cohort with no separate UI control needed, exactly the "natural flow" the locked spec called for. The Cohorts detail view gained a REPLAY badge (list and detail), the survivorship note, an equity-curve chart (hand-rolled SVG, the same coordinate-mapping pattern the existing chart modal and evidence sparklines use), the metrics grid, a contributors table (now including each symbol's own max drawdown), and an "evaluate to" date control with its own apply/reset-to-latest affordance. `python -m screener.cli cohort create --as-of YYYY-MM-DD [...]` freezes a replay cohort from the CLI; `cohort perf <id> [--end DATE]` prints the same panel as readable text.

**Verification.** 21 new tests in `tests/test_cohort_perf.py` (exceeds the ROADMAP's ≥12 target) covering mode auto-set and its non-overridability, the OOS-scorecard wall, back-compat defaulting for pre-Item-17 records (old cohort files have no `mode`/`as_of` fields at all — `_load_all()` backfills both on read), window resolution (pending semantics, clamping, non-trading-day rollback), hand-computed metrics against an engineered deterministic price path (expected values computed independently in-test, not hardcoded literals — max drawdown, equity-curve indexing, baseline excess), the 60-bar Sharpe gate on both sides, weighted-vs-equal contribution arithmetic, stale-symbol handling within a window, and adjustment invariance for a replay window (the halving test, reused at a historical as_of). 8 more in `tests/test_evaluator.py`'s `TestCohortReplayAndPerformanceEndpoints` cover the API surface, plus a `survivorship_note` wiring test added after the gap above was caught.

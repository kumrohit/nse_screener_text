## 1. Purpose and scope

A screener over the Nifty 500 universe that accepts filters written in plain English ("stocks taking support at 50 EMA and in an uptrend"), compiles them into a deterministic filter specification, and evaluates that specification against five years of daily price and volume history. Version scope is deliberately limited to price/volume technical analysis: no fundamentals, no intraday, no derivatives data. The parser is required to refuse anything outside this scope rather than approximate it.

## 2. Design principles

The system is built around one central rule: **natural language never touches computation.** The LLM's sole responsibility is translating text into a validated JSON specification (the DSL). Everything downstream ‚Äî indicators, condition evaluation, result assembly ‚Äî is deterministic, pure, and unit-testable. Consequences of this rule:

1. Every screen is reproducible: the compiled DSL plus the data snapshot date fully determine the output.
2. Ambiguity is resolved once, in a documented canonical vocabulary (¬ß7), not per-query by model whim.
3. The user always sees the compiled interpretation in plain English before results, and can bypass the LLM entirely with `--json`.
4. The parser must return an explicit error for unmappable phrases. Guessing is a defect, and the golden-query suite (¬ß10) treats it as one.

Secondary principles: fail loud (stale data, unknown fields, and missing benchmarks raise, never silently return empty results); no look-ahead anywhere (pivots need confirmation bars, historical as-of screens only use data up to the as-of row); prefer auditable pure-pandas formulas over indicator libraries.

## 4. Data layer

**Universe.** Official Nifty 500 constituent CSV from NSE archives, fetched with browser-like headers (NSE blocks default user agents and most datacenter IPs), cached to `data/nifty500.csv`. On fetch failure the cached copy is used with a warning; with no cache the system refuses to run and tells the user how to obtain the file manually. The universe drifts with semi-annual index rebalances ‚Äî re-fetch with `force_refresh=True` quarterly.

**Prices.** yfinance daily bars with `.NS` suffix, five years of history, `auto_adjust=True`. Auto-adjustment folds in splits and bonuses (essential ‚Äî unadjusted data destroys every moving average at each corporate action) and also dividends (a minor distortion of raw levels, acceptable for technical work and strictly better than the alternative). Downloads run in chunks of 50 tickers with three retries and exponential-ish backoff, plus a two-second inter-chunk sleep, because yfinance throttling is the dominant operational risk of the free-data route. Storage is long-format Parquet (`symbol, date, open, high, low, close, volume`), roughly 600k rows for the full universe.

**Cleaning.** Rows are dropped when any OHLC value is non-positive or high < low. Timestamps are normalised to tz-naive dates.

**Incremental updates.** `update` re-fetches a ten-day tail for all symbols and merges with `keep="last"`, absorbing Yahoo's occasional late corrections. Schedule after ~18:30 IST when NSE close data settles on Yahoo.

**Benchmark.** Nifty 50 (`^NSEI`) close series stored separately in `benchmark.parquet`, refreshed on every backfill/update, consumed only by the `rel_strength` condition (which fails loud if the benchmark is absent).

**Integrity gates before any screen runs:** staleness check (latest bar within 5 calendar days, else refuse with instructions), minimum history (symbols with fewer than 60 bars are excluded from panels), and a liquidity gate (20-day median turnover ‚â• ‚Çπ0.5 cr ‚Äî mostly redundant for Nifty 500 but guards against data glitches surfacing dead tickers).

**Known data risks and mitigations.** (a) NSE symbol ‚âÝ Yahoo ticker for a handful of names post merger/rename; the ingester skips failures silently, so compare the backfilled symbol count against 500 after the first run and maintain an alias map in `universe.py` if needed. (b) yfinance API changes break periodically; the ingest module is the designated swap point ‚Äî replacing it with NSE bhavcopy ingestion touches nothing else. (c) Survivorship: the store only contains current constituents, so historical as-of screens are biased toward survivors. Acceptable for a discovery tool; documented so nobody mistakes Phase 3 screen-backtests for survivorship-clean research. (d) yfinance adjustment-splice bug: for at least three symbols (CGCL, GPIL, MOTILALOFS), `auto_adjust=True` applies the correct split/bonus ratio but splices it in at the wrong date ‚Äî a large jump appears on 2024-01-01 for all three, months before their true record dates (Mar/Jun/Oct 2024). Confirmed via `verify --jumps`: `refetch` returns byte-identical data, so this is upstream in Yahoo's own adjusted series, not a local staleness issue. Harmless for live screens once the affected window ages out of the longest lookback (252 bars), but historical as-of replays dated within the corrupted window for those symbols will show wrong EMA/RSI/52-week values. No general fix; flagged here so it isn't mistaken for a fixable ingestion bug.

## 4a. Data layer v2 ‚Äî NSE bhavcopy (ROADMAP Item 3, pre-cutover)

`screener/bhavcopy.py` runs **side-by-side** with the yfinance store above ‚Äî its own store (`data/bhavcopy_prices.parquet`), read by nothing in the evaluator/webapp/CLI screen path yet. Cutover happens only after ~2 weeks of evidence from the cross-source check below.

**Source.** NSE's daily `sec_bhavdata_full_DDMMYYYY.csv` ‚Äî confirmed by fetching a live file (2026-07-03) rather than assumed from training data: one file has OHLCV *and* delivery % together (`SYMBOL, SERIES, DATE1, ..., CLOSE_PRICE, ..., TTL_TRD_QNTY, ..., DELIV_QTY, DELIV_PER`), filtered to `SERIES == "EQ"`. A separate delivery-only file turned out to be unnecessary. `bhavcopy.update_bhavcopy_store()` fetches missing business days since the store's last date (or the last 10 calendar days on first run); a 404 is treated as a weekend/holiday, not a failure ‚Äî NSE's own calendar is the source of truth. First real run (2026-07-05) pulled 2,431 symbols (all NSE EQ series, not just Nifty 500), 14,403 rows, 2026-06-25 ‚Üí 2026-07-03.

**Acceptance check, run for real:** bhavcopy closes vs the yfinance store for the same date (2026-07-03), all 500 Nifty 500 symbols ‚Äî median absolute difference ~2√ó10‚Åª‚Å∂%, max ~5√ó10‚Åª‚Å∂% (floating-point rounding only), zero symbols over 0.5%. Both sources are unadjusted for that single day (adjustment convention only affects *historical* bars), so this is a clean apples-to-apples check.

**Prices are raw/unadjusted** from both NSE files ‚Äî same as yfinance without `auto_adjust`. Our own adjustment pipeline: `fetch_corporate_actions()` hits `/api/corporates-corporateActions` (needs a cookie warm-up `GET` to nseindia.com first; that warm-up itself may 403 yet still set the cookies the API call needs, so its status isn't checked). `parse_adjustment_factor(subject)` extracts a multiplicative factor from the free-text `subject` line using two regexes built from **real** NSE text (fetched 2026-07-05, Jan 2024‚ÄìJul 2026 window), not guessed formats:
- `"Bonus X:Y"` (X new shares per Y held) ‚Üí `factor = Y / (X + Y)`
- `"Face Value Split (Sub-Division) - From RsA/- Per Share To RsB/- Per Share"` ‚Üí `factor = B / A`

Splits/bonuses only ‚Äî dividends and anything else return `None` and are excluded from adjustment (documented divergence from yfinance's `auto_adjust`, which folds dividends in too). CGCL is a good cross-check: NSE lists *two* separate 2024 actions for it (a 1:1 bonus **and** a 2‚Üí1 face-value split), and `0.5 √ó 0.5 = 0.25` matches the real jump ratio investigated in ROADMAP Item 0 almost exactly. `build_adjustment_factors()` compounds each symbol's factors going backward in time (a bar's cumulative factor is the product of its own action's factor and every *later* action's factor ‚Äî the same convention `auto_adjust` uses); `apply_adjustments()` must apply actions **latest-ex-date-first**, since an earlier action's `date < ex_date` mask is always a subset of every later action's mask ‚Äî applying oldest-first would let the later (broader, single-factor) write clobber the earlier (narrower, correctly-larger cumulative-factor) one. Caught and fixed during implementation; regression-tested (`TestAdjustmentCompounding.test_apply_adjustments_ordering`).

**Cross-source consistency check** (`verify.check_cross_source`, folded into `python -m screener.cli verify` automatically): compares overlapping `(symbol, date)` closes between the two stores and WARNs (not FAILs ‚Äî bhavcopy isn't consumed by any screen yet) above a 0.5% threshold. First real run found 45/3,000 overlapping bars over threshold, concentrated in 11 symbols (e.g. UNIONBANK ~2.95%, TECHM ~2.60%) ‚Äî but the gap is *constant* across every date for a given symbol, the signature of a past dividend adjustment already folded into yfinance's series (not a bug; exactly the divergence source documented in ¬ß4(d) above). A systematic, non-constant gap would be the signal to actually investigate before counting a day toward the 2-week cutover evidence window.

**Not yet done** (explicitly gated): the `delivery` DSL condition, its parser vocabulary, and the "accumulation" preset ‚Äî ROADMAP marks these "only after cutover," so they aren't wired into `dsl.py`/`evaluator.py`/`parser.py` yet even though `delivery_pct` already exists in the bhavcopy store. Also not done: the config-flag cutover itself, and adding `bhavcopy-update` to the nightly cron (currently a manual command ‚Äî add it alongside `update` once the side-by-side evidence period is underway).

## 5. Indicator engine

Pure pandas/numpy; every formula lives in `indicators.py` and is auditable. Wilder-style indicators (RSI, ATR, ADX) use `ewm(alpha=1/n)` smoothing, matching standard charting-platform values. EMAs use `span=n, adjust=False` with `min_periods=n` so warm-up rows are NaN rather than misleading ‚Äî and every condition treats NaN as False.

Daily panel columns: OHLCV; EMA 10/20/50/100/200 each with a 5-bar difference slope; SMA 20/50/150/200 each with a 5-bar difference slope (150 and its slope added in v0.9 for the Minervini "stage 2" template, ¬ß12c); RSI(14); ATR(14) and ATR%; ADX/+DI/‚àíDI(14); MACD(12,26,9) line/signal/histogram; Bollinger(20, 2œÉ) upper/lower/width%; 20-day average volume and volume ratio; turnover in ‚Çπ crore; 252-bar rolling 52-week high/low and percentage distances; rate of change over 5/21/63/126/252 bars (‚âà 1 week / 1 month / 1 quarter / 6 months / 12 months); `mom_12_1` ‚Äî the 12-1 skip-month momentum return from 252 bars ago to 21 bars ago (Jegadeesh & Titman 1993 convention, ¬ß12c) ‚Äî deliberately distinct from `roc_252`, which includes the most recent month.

Weekly panel (computed lazily, only when a screen contains a weekly condition): W-FRI resample of OHLCV; EMA 10/20/40 with 3-week slopes; RSI(14); ROC over 4 and 13 weeks. The in-progress week appears as a partial bar ‚Äî a deliberate choice so "weekly uptrend" reflects the current state rather than last Friday's, at the cost of the last weekly bar being mutable until Friday's close.

Trend is defined canonically, not left to interpretation. Daily uptrend: close > EMA50 > EMA200 and EMA50 5-bar slope > 0. Weekly uptrend: close > EMA20w > EMA40w and EMA20w 3-week slope > 0. Downtrends are exact mirrors.

## 6. Filter DSL

A screen is `{"logic": "AND"|"OR", "conditions": [...], "as_of": "latest"|ISO-date}`. Validation is strict: unknown condition types, unknown fields, bad operators, and weekly-timeframe use of daily-only fields all raise `DSLValidationError` before any data is touched. Twenty-one condition types:

| Type | Keys (defaults) | Semantics |
|---|---|---|
| `compare` | left, op, right | Field vs field-or-number at the as-of row. |
| `proximity` | target, ref, tolerance_pct, lookback (3) | \|target‚àíref\|/ref ‚â§ tol on any bar in the window. |
| `trend` | direction | Canonical trend definition (¬ß5). |
| `support_at_ma` | ma, tolerance_pct (1.5), lookback (3) | See below. |
| `cross` | fast, slow, direction, lookback (3) | Sign change of fast‚àíslow within the window. |
| `volume_spike` | min_ratio | volume / 20-day average ‚â• ratio. |
| `range` | field, min and/or max | Inclusive band check. |
| `change` | field, window, op, value_pct | Percent change over window bars. |
| `near_support` | tolerance_pct | Close within tol above nearest swing support (¬ß8). |
| `near_resistance` | tolerance_pct | Close within tol below nearest swing resistance. |
| `breakout_resistance` | lookback (5), buffer_pct (0) | See ¬ß8. |
| `rel_strength` | window, op, value_pct | Stock return ‚àí Nifty return over window, in pct points. |
| `candle` | pattern, lookback (1) | One of 6 exact candlestick formulas (¬ß9a) within the window. |
| `tight_range` | max_range_pct, bars (10) | N-bar high-low span as % of the window low. |
| `bb_squeeze` | percentile (20), lookback (252) | Bollinger bandwidth vs its own trailing distribution. |
| `flat_base` | bars (20), max_range_pct (12), max_from_52w_high_pct (15) | Tight range near the 52-week high (¬ß9a). |
| `sector` | in (list of exact industry strings) | Stock's Nifty 500 industry classification is one of the given sectors. |
| `rs_percentile` | window (63), basis ("return"), op, value | Stock ranked as a percentile (0‚Äì100) across the universe on the as-of date (¬ß6a). `basis="return"`: own window-bar return. `basis="mom_12_1"`: the 12-1 skip-month construction (¬ß12c); `window` is ignored. |
| `sector_rank` | window (63), top **xor** bottom | Stock's sector is among the top/bottom N sectors by equal-weight window-bar momentum (¬ß6a). |
| `gap` | direction, min_gap_pct (2.0), lookback (3) | Open vs the prior bar's close, on any bar within the window. |
| `atr_pct_percentile` | op, value | Stock's ATR% (volatility) ranked as a percentile (0‚Äì100) across the universe; 0 = least volatile (¬ß12c). |

`compare`, `range`, `trend`, `change`, and `cross` accept `"timeframe": "weekly"`, which routes evaluation to the weekly panel with its restricted field set.

### 6a. Cross-sectional pre-pass

`rs_percentile`, `sector_rank`, and (v0.9) `atr_pct_percentile` need context beyond a single symbol's panel, so `run_screen` (and webapp's `/api/screen`) compute a **cross-sectional pre-pass** once per screen ‚Äî not once per symbol ‚Äî before the per-symbol loop: `cross_section.build_cross_section(panels, universe, as_of, window)` returns a table indexed by symbol with that window's return, its RS percentile, the equal-weight mean return of its sector, the sector's momentum rank (1 = best), the panel's `mom_12_1` value and its percentile (v0.9, ¬ß12c), and the panel's `atr_pct` value and its percentile (v0.9, ¬ß12c ‚Äî ranked ascending, so 0 = least volatile in the universe on that date). This is the one structural change Item 2 required: `evaluate_symbol`/`explain_symbol` gained `symbol`, `sector_by_symbol`, and `cross_section` parameters (all optional, default `None`, so every existing call site keeps working unchanged). `atr_pct_percentile` and `mom_12_1`-basis `rs_percentile` share the same window-keyed cache as `rs_percentile`/`sector_rank` even though their ranking doesn't depend on the window value ‚Äî cheap to compute redundantly, and it lets a screen combining them at the same window reuse one cache entry.

Design points: pure function, in-process cache only keyed by `(id(panels), as_of, window)` ‚Äî no disk state, cheap enough (single-digit milliseconds over 500 symbols √ó 5y; measured well under the 5s budget) that a persistent cache isn't worth the invalidation complexity in v1. Thin-history symbols (fewer than `window` bars up to `as_of`) get `ret_pct = NaN` and are excluded from `rank(pct=True)` rather than defaulted to the 0th/100th percentile ‚Äî pandas keeps NaN as NaN through ranking, so this falls out of the implementation rather than needing a special case. Historical as-of correctness: every value is recomputed from the row at or before `as_of`, never sliced from "latest" ‚Äî verified by a dedicated look-ahead test (same spirit as the pivot-confirmation test in ¬ß8): two symbols whose relative sector performance flips between an early date and today rank in opposite order at each date.

`sector_rank`'s `bottom` (the mirror of `top`, for "laggard" screens) is a controlled extension beyond the two DSL condition shapes literally spec'd for Item 2 ‚Äî `top`/`bottom` are mutually exclusive, validated the same way `trend`'s up/down and `cross`'s above/below are.

**"Taking support at a moving average"** ‚Äî the phrase that motivated the whole design ‚Äî is defined as: within the lookback window, (a) the low touched the MA (came within tolerance, or pierced below it), (b) the close held at or above MA √ó (1 ‚àí tol) on *every* bar of the window, and (c) the latest close is strictly above the MA. Condition (b) is what separates a bounce from a breakdown: a stock that slices through the MA and closes 3% below it has touched the MA but not taken support. Condition (c) requires the bounce to have actually happened, excluding stocks still sitting on the level.

**Tolerance calibration (2026-07-05):** ran `support_50ema_uptrend` across four as-of dates (2026-04-01, 05-01, 06-01, 07-03), 3 matches each. Decision: keep the 1.5% / 3-bar default. Revisit if a larger sample or the near-miss lists suggest it's too tight or too loose in live use.

The `describe()` function renders any validated spec back to plain English; the CLI prints this before every run and `--dry-run` stops there. This is the primary trust mechanism.

## 7. NL parser

Single Anthropic API call (`claude-sonnet-4-6`, temperature default) with a system prompt containing the full schema, the exhaustive allowed-field list, and the canonical vocabulary. Key mappings: uptrend/downtrend ‚Üí `trend`; "taking support at / bouncing off <MA>" ‚Üí `support_at_ma` (the user naming an MA distinguishes this from horizontal `near_support`); "near X" ‚Üí `proximity` tol 2.0; golden/death cross ‚Üí EMA50/EMA200 cross, lookback 5; volume spike ‚Üí ratio 1.5, "huge/massive" ‚Üí 2.5; oversold/overbought ‚Üí RSI ‚â§30 / ‚â•70; "near 52-week high" ‚Üí distance ‚â• ‚àí5%; "strong trend" ‚Üí ADX ‚â• 25; "outperforming the Nifty" ‚Üí `rel_strength` 63-bar > 0; "weekly uptrend" ‚Üí weekly trend condition; "<sector> stocks"/"in the <sector> sector" ‚Üí `sector` (common short forms like "IT" or "pharma" map to the exact industry string ‚Äî see ¬ß6a ‚Äî but only when the prompt's fixed sector list has an unambiguous match); "RS above N" ‚Üí `rs_percentile` window 63, op ‚â•; "market leaders"/"top sector" ‚Üí `sector_rank` top 3; "lagging sector" ‚Üí `sector_rank` bottom 3. Time words map as 1 week = 5 bars, 1 month = 21, 3 months = 63 (weekly: 4 and 13).

Two traps are encoded explicitly in the prompt. First, "breakout above 52-week high" must **not** compile to `close > high_52w` ‚Äî the rolling high includes today, so that comparison is nearly unsatisfiable; it maps to `pct_from_52w_high ‚â• ‚àí0.5` instead. Second, bare "moving average" defaults to EMA; "DMA" or "simple" selects SMA.

Hard parser rules: output raw JSON only; never invent thresholds beyond canonical defaults unless the user states them; AND unless the user says or/either; return `{"error": "..."}` for anything unmappable (fundamentals, news, intraday, or a sector adjective that doesn't clearly match one of the fixed industry strings). Parser output still passes through `dsl.validate()` ‚Äî the LLM is never trusted to be well-formed.

## 8. Swing support/resistance methodology

Pivot detection uses the fractal method: bar j is a pivot high when its high is the maximum of bars [j‚àík, j+k], k = 5 (mirror for lows). The final k bars of any window can never confirm a pivot because the right side of the fractal hasn't printed ‚Äî this is the standard confirmation lag and doubles as the look-ahead guard, verified by a dedicated test.

Levels are built per symbol, per as-of row, from pivots in the trailing 250 bars: pivot prices are sorted and greedily clustered (a price joins the running cluster if within 1.0% of the cluster's last member); each cluster's mean is a candidate level with a touch count; levels with fewer than 2 touches are discarded as noise. Nearest support is the highest surviving level at or below the close; nearest resistance is the lowest above it.

`breakout_resistance` computes levels **as of `lookback` bars ago** (so the breakout move itself cannot create or erase the level being broken), takes the nearest resistance from that vantage point, and requires the current close to exceed it by the optional buffer. This is intentionally conservative; the documented limitation is that very recent consolidations (younger than pivot confirmation + lookback) can't produce breakout signals yet.

Parameter choices (k=5, 250-bar lookback, 1% cluster tolerance, 2-touch minimum) are defensible defaults, not optimised values ‚Äî they live in `sr.py` constants and are candidates for sensitivity analysis in Phase 3.

## 9a. Pattern definitions

Candlestick and consolidation conditions use these exact formulas (bar j; body = |close‚àíopen|, range = high‚àílow, wicks measured from body edges). **Inside bar**: high < previous high AND low > previous low. **NR7**: range is the strict minimum of the last 7 bars. **Bullish engulfing**: previous bar red, current bar green, current open ‚â§ previous close and current close ‚â• previous open (bodies only; wicks ignored). **Bearish engulfing**: exact mirror. **Hammer**: lower wick ‚â• 2√ó body AND upper wick ‚â§ 30% of range. **Shooting star**: mirror. `candle` conditions accept a lookback (default 1 = latest bar only). **Tight range**: (max high ‚àí min low) / min low over N bars ‚â§ threshold. **BB squeeze**: current Bollinger bandwidth ‚â§ the given percentile of its own trailing 252-bar distribution (‚â•60 observations required, else fails closed). **Flat base**: tight range over 20 bars (‚â§12%) AND close within 15% of the 52-week high ‚Äî the pre-breakout base structure. These definitions are deliberately strict and singular; trader phrases map to them via the canonical vocabulary or not at all.

A curated preset library (`presets.py`, 26 named screens grouped by intent ‚Äî trend continuation, breakouts, reversals, relative strength, bearish) exposes ready-made specs via `GET /api/presets` (web dropdown), `screener presets`, and `screener screen --preset <id>`. Every preset is validated at import time, so DSL changes that break a preset fail the test suite immediately. Since v0.9 (¬ß12c), every preset also carries an `evidence` object pointing back to `LITERATURE.md`.

## 9. Evaluator semantics

Conditions are pure functions `(panel, condition, row_index) ‚Üí bool`. NaN anywhere in a condition's inputs yields False (a stock with insufficient history cannot match ‚Äî it is never accidentally included). `as_of` resolves to the last row at or before the given date, independently for daily and weekly panels. `logic` applies flat AND/OR across conditions; nested boolean trees are explicitly out of scope for v1 (the parser refuses queries requiring them). The runner applies the liquidity gate, evaluates each symbol, and emits a result row of screening context: close, % vs EMA50, RSI, volume ratio, 1M/3M returns, distance from 52-week high, plus name and industry from the universe file, sorted by 3-month return.

## 10. Testing strategy

Three layers. **Synthetic-series unit tests** (236 passing, incl. 21 in `tests/test_allocate.py` for the allocation engine ‚Äî risk/sector/capital cap invariants, degenerate cases, method divergence on engineered volatility dispersion, and a regression test for the aggregate-capital-cap bug caught via live testing; ## 11. Operations runbook

First run: create a Python 3.10+ venv (`python3.12 -m venv .venv && source .venv/bin/activate` ‚Äî both entry points guard against older interpreters and print this fix), `pip install -r requirements.txt`, set `ANTHROPIC_API_KEY`, `python -m screener.cli backfill` (10‚Äì15 min), then `python -m screener.cli verify` ‚Äî an automated 13-check health report (coverage vs index list, freshness, depth, bar integrity, duplicates, corporate-action smell test via >40% single-day move counts, benchmark presence, RSI/EMA spot checks on a 25-symbol sample, the data-layer-v2 cross-source check ¬ß4a, and screen-log JSONL integrity) that exits non-zero on FAIL for pipeline gating; `verify --jumps` lists the exact bars behind the adjustment smell test with split-ratio hints, and `refetch SYMBOL` drops and freshly re-downloads one symbol as the unadjusted-data remedy. Nightly: cron `python -m screener.cli update` after 18:30 IST (add `bhavcopy-update` alongside it once side-by-side collection should run unattended ‚Äî see ¬ß4a). Quarterly: refresh the universe and re-backfill to pick up rebalances. Screens: `screen "<text>"`, `--dry-run` to inspect the compiled spec, `--json` to bypass the LLM, `--as-of YYYY-MM-DD` to replay a historical date (CLI parity with the web UI's date picker), `--out file.csv` to export. Backtests: `backtest "<text>"` (or `--preset id` / `--json`) runs the same spec as an event study ‚Äî `--horizons`, `--cooldown`, `--cost-pct`, `--stride`, `--hypothesis`, `--no-sensitivity` ‚Äî a readable text report with the survivorship caveat always printed last. `--universe` is accepted on `backfill`/`update`/`verify`/`screen`/`backtest` (¬ß12g) ‚Äî `nifty500` (default) or `nse_full`; the webapp has an equivalent header dropdown. Failure modes: stale-store error ‚Üí run update; universe fetch fails ‚Üí cached list used automatically, manual CSV drop-in as last resort; yfinance thin/missing data for a symbol ‚Üí excluded by the 60-bar minimum, check the alias problem in ¬ß4; parser refusal ‚Üí the query needs rephrasing within scope, or the concept belongs on the roadmap.


## 12. Web UI and evidence layer

`python -m screener.webapp` serves a single-page interface (FastAPI backend, dependency-free vanilla-JS frontend in `web/index.html`, port 8501) built around full auditability of every screen. The page always shows, together: the original query, the plain-English compiled interpretation, the raw JSON spec (collapsible), the data as-of date and mode, and funnel statistics (universe ‚Üí liquidity-excluded ‚Üí evaluated ‚Üí matched).

**Screen definition** offers three routes: plain-English (LLM parse with mandatory interpretation echo), raw JSON spec, and a grouped **preset dropdown** backed by `presets.py` (¬ß9a) via `GET /api/presets`, which shows each preset's rationale and compiled English before running. An **as-of date picker** replays any historical date; when set, condition evaluation, match metrics, and sparkline windows are all computed at that row (never the latest bar), preserving the reproducibility guarantee for historical screens. Note the survivorship caveat from ¬ß4 applies to any historical replay.

The core of the UI is the **evidence trail**, powered by `explain.py`. For every condition on every displayed stock it reports pass/fail plus the observed values behind the decision ‚Äî e.g. for `support_at_ma`: the date the low touched the MA and the touch distance, the worst close-vs-MA excursion across the window, and the latest close's margin above the MA. Pass/fail is delegated to the same `evaluator.cond_*` functions used by the screen itself, so the explanation can never disagree with the result; `explain.py` only adds observability. Each match card also renders an **evidence sparkline**: an inline SVG of the last 60 bars (up to the as-of row) overlaying exactly the series the spec referenced (EMAs, bands, 52-week levels) and every horizontal level the evidence produced (swing support/resistance, breakout levels) ‚Äî so verifying a touch or a level does not require leaving the page. A **near-miss** section lists stocks failing exactly one condition with the failing condition marked ‚úó ‚Äî the most useful feedback for tuning tolerances; a toggle button hides/shows it without losing the funnel-stat count. Match cards are capped at `MAX_MATCHES` (100) in the payload ‚Äî `stats.matched` always reports the true total, so a loose filter's size is never hidden, just not rendered as hundreds of DOM cards; a note above the results explains the cap when it's active. A client-side **CSV export** button downloads the currently-displayed matches (symbol, name, industry, and every snapshot metric) without a dedicated export endpoint, since the data is already in the page.

Every run is appended to `data/screen_log.jsonl` ‚Äî timestamp, as-of date, full spec, its plain-English description, funnel stats, matched symbols ‚Äî turning the deterministic-replay guarantee into a browsable history (`GET /api/log`, or `python -m screener.cli log`). Log writes never raise; a failed write cannot break a screen. A **recent screens** panel in the UI fetches this log and lets you replay any past run with one click (restores the spec into the JSON tab and the as-of date), refetching whenever a new screen completes so it never shows stale entries.

Endpoints: `GET /api/status`, `GET /api/health` (¬ß12a), `GET /api/presets`, `GET /api/log`, `POST /api/parse` (text ‚Üí spec via the LLM; structured error without an API key), `POST /api/screen` (spec ‚Üí matches + near-misses + sparklines + methodology block). When `data/prices.parquet` is absent the backend boots a labelled synthetic 11-stock demo universe (`demo.py`) with engineered behaviours ‚Äî including three exemplars whose only purpose is exercising the data-quality flags in ¬ß12a ‚Äî so the UI, API, and evidence layer are fully testable ‚Äî in CI and by a fresh clone ‚Äî without market data.

## 12a. Operational robustness (v0.7 Item 6)

**Stale-server fix (P0).** `webapp._load_state()` used to load panels once at process startup and cache them forever ‚Äî a long-running server would silently keep screening yesterday's data after the nightly `python -m screener.cli update` wrote a fresh `prices.parquet`. It now compares the store's mtime on every call and rebuilds (clearing the cross-sectional pre-pass cache too, since it's keyed by `id(panels)` and a GC-reused id could otherwise serve stale ranks) whenever the file has changed underneath it. Verified with a test that swaps the underlying parquet file mid-session and confirms `as_of` updates without a restart ‚Äî not just that the code compiles.

**Data-quality flags.** Every match/near-miss now carries a `flags` list computed from signals `verify`'s checks already look for, but scoped to one symbol and surfaced where a user is actually looking: `jump` (a >40% single-day move within the spark window ‚Äî possible unadjusted split/demerger, "levels may straddle a gap"), `thin_history` (fewer than 250 bars as of the as-of row ‚Äî long-lookback indicators unreliable), `stale` (the symbol's last bar predates the store's overall latest ‚Äî possibly suspended/delisted). The UI renders a small `‚öÝ N data flag(s)` badge with the reasons in a tooltip. `demo.py`'s JUMPY/THINHIST/STALECO exist solely to exercise these; live-tested against the real store too ‚Äî it correctly flagged a genuine recently-listed Nifty 500 name as `thin_history`.

**Config overrides.** `config.py` loads an optional `data/config_local.toml` at import time, overriding any of a fixed allowlist (`_OVERRIDABLE`): the liquidity gate, staleness window, the four SR pivot/clustering constants (moved from `sr.py` into `config.py` so overrides actually take effect ‚Äî `sr.py` keeps them as module aliases for compatibility), and the web UI's spark-bar count / match cap. Unknown keys in the file are flagged and ignored, never silently creating an unused setting. `config.config_hash()` ‚Äî a short hash of every overridable tunable's effective value ‚Äî is logged with each screen-log entry and shown in the methodology footer, so a screen's result is reproducible together with the config that produced it, not the spec alone. Requires `tomli` on Python <3.11 (stdlib `tomllib` on 3.11+).

**Parser resilience.** `parser.parse_with_assumptions()` retries once, with a corrective instruction, if the LLM's response isn't valid JSON ‚Äî a prompt-following failure, not a scope refusal. If both attempts fail, or the resulting spec fails `dsl.validate()`, the query and raw output are appended to `data/parse_failures.jsonl` as a vocabulary-improvement backlog; legitimate `{"error": ...}` scope refusals are deliberately *not* logged there, since that's the parser working as intended. The LLM may optionally return a top-level `"assumptions"` list naming any canonical default it filled in because the user gave no explicit number ‚Äî stripped from the spec before validation, surfaced by `/api/parse` and rendered in the UI as "interpreted with defaults: ‚Ä¶". `parser.parse()` remains a thin backward-compatible wrapper returning just the spec.

**`/api/health`.** Cheap JSON for cron/uptime monitoring, cheap enough to curl after every nightly `update`: mode, as-of, store mtime (confirms the P0 fix actually picked up the fresh file), panel count, benchmark presence, log writability, `git describe --always --dirty` as the deployed version, and the config hash.

**Screen-log rotation.** Once `data/screen_log.jsonl` exceeds 5,000 lines, the oldest entries move into `data/screen_log.rotated.jsonl` rather than growing the active file forever or discarding history. `verify.check_screen_log()` validates both files' JSONL integrity together.

**Golden harness in CI.** `.github/workflows/golden-harness.yml` ‚Äî `workflow_dispatch` only (needs an `ANTHROPIC_API_KEY` repo secret to hit the real API), so a parser-prompt PR can be gated on 19/19 without requiring a local run. The existing `ci.yml` push/PR job remains offline-only.

## 12b. UI depth (v0.7 Item 5) ‚Äî daily-driver features

**Screen diff ("what changed since last run").** `dsl.spec_hash()` ‚Äî moved here from what used to be golden_harness-only canonicalisation logic (`dsl.canonicalize_conditions`, now shared by both) ‚Äî hashes a screen's logic + conditions (defaults filled, sorted, **as_of excluded**: rerunning the same criteria on newer data is still "the same screen"). Every screen-log entry now carries its `spec_hash`; `/api/screen` looks up the most recent prior entry with a matching hash, diffs the matched-symbol sets, and re-explains each dropped symbol against *current* data so the UI shows exactly which condition now fails rather than just that a symbol vanished. Surfaced as a "+N new / ‚àíM dropped" badge with an expandable dropped list; new matches get a small NEW tag on their card.

**Full chart modal.** `_referenced_fields`/`_evidence_levels` were factored out of the existing inline-spark builder so a new `_chart_payload` (CHART_BARS=250, full OHLCV) could reuse them without duplicating the overlay logic. `POST /api/chart` fetches one symbol's full window lazily on click ‚Äî embedding 250 bars per match in every `/api/screen` response would have bloated the main payload for no benefit, since most matches are never opened. The frontend renders hand-rolled SVG candlesticks (no external charting library ‚Äî the offline-first constraint from ¬ß1 stands) with a volume subpane and drag-to-zoom (mouse down/up over the SVG re-renders a narrower `[start,end]` bar range client-side, no server round-trip).

The frontend merges these into the same `PRESETS` array the built-in dropdown already uses (`id` prefixed `user:` to avoid collisions), so the existing preset-selection code path needed no changes ‚Äî just a "My screens" optgroup and a small manage-panel for rename/delete.

**Multi-screen dashboard.** `_run_screen(spec)` is the extraction that made this cheap: it's the entire body of what `/api/screen` used to inline, now callable per-preset. `POST /api/screen_batch` runs N built-in-or-saved presets through it and returns just (matched count, top-3 symbols by 3-month return, new-since-last-run) per screen ‚Äî the existing diff/log machinery just works, since each batch entry is a real screen run in its own right. ## 12c. Evidence-based strategy presets (v0.9 Item 9)

**`LITERATURE.md` came first.** Per the roadmap's explicit sequencing, the review document was written before any preset or indicator it justifies ‚Äî a preset that can't point to a section there doesn't ship as evidence-based. It covers eight price/volume anomaly families (cross-sectional momentum, 52-week-high anchoring, time-series momentum, moving-average rules, volume-confirmed momentum, the low-volatility anomaly, the practitioner Minervini/O'Neil "stage 2" template, and consolidation breakouts), each with the canonical papers, the core finding, India-specific evidence where it exists, and an honest caveat ‚Äî including, deliberately, the ones that undercut the family's own tradability (momentum crashes, the Sullivan-Timmermann-White data-snooping critique of moving-average rules, survivorship bias in practitioner track records).

**Seven new presets**, each pointing at a specific LITERATURE.md section: `momentum_12_1_leaders` (¬ß1, academic), `near_52w_high_ghw` (¬ß2, academic), `tsmom_regime` (¬ß3, academic), `ma_timing_highvol` (¬ß4, mixed ‚Äî MA rules restricted to the high-volatility tercile where Han-Yang-Zhou found the effect concentrated), `volume_momentum` (¬ß5, academic), `lowvol_defensive` (¬ß6, academic), and `minervini_stage2` (¬ß7, explicitly practitioner ‚Äî labeled as such, not dressed up). All 19 pre-existing presets were annotated retroactively rather than left unlabeled; several map only loosely to the eight vetted families (candlestick patterns, sector-rank combinations) and are honestly marked `"practitioner"`/`"mixed"` with a finding stating no dedicated study was reviewed, rather than a fabricated citation.

## 12d. Portfolio allocation engine (v0.10 Item 10)

`screener/allocate.py` turns a ranked list of already-screened symbols + capital + a risk tolerance into integer-share position sizes. It is explicitly framed ‚Äî in this module's docstring, the API response (`disclaimer` field), and the UI ‚Äî as a **sizing calculator with documented methodology, not a recommendation engine**: it has no view on which stocks to buy, only on how much of each. Non-goals, deliberate: no mean-variance optimisation (estimation error dominates on small screened subsets ‚Äî DeMiguel, Garlappi & Uppal 2009), no Kelly criterion (drawdown profile unsuitable for discretionary use), no return forecasts, no auto-execution.

**Three methods**, always computed against the same caller-ranked symbol list (this module never re-ranks ‚Äî the caller, e.g. the UI's current results sort, decides priority when candidates exceed the caps):
- **`risk`** (default) ‚Äî fixed-fractional risk sizing (Van Tharp/Turtle-style): per-position risk = capital √ó risk_pct%, stop distance = 2√óATR(14), shares = floor(risk √∑ stop distance).
- **`inverse_vol`** ‚Äî naive risk parity: capital weight ‚àù 1/ATR% (calmer names get more capital).
- **`equal`** ‚Äî 1/N. Computed and returned as an always-present `baseline` key alongside whichever method the caller actually requested (unless they requested `equal` itself) ‚Äî DeMiguel et al found 1/N is the benchmark no optimiser reliably beats out-of-sample on estimated inputs, so it's shown for comparison rather than buried as just another dropdown choice.

**Constraints applied to every method, in this order**: per-position cap (`max_position_pct`, default 15% of capital), sector cap (`sector_cap_pct`, default 30% of capital per industry, enforced by tracking running per-sector deployed value and trimming/excluding later same-sector candidates), and ‚Äî the aggregate capital cap ‚Äî a running `capital_deployed` total that trims or excludes later candidates once the sum would exceed the input capital. **This last cap was missing in the first implementation and caught only via live testing against the real 500-symbol store**: nine candidates each individually satisfying the 15% position cap and passing distinct sector caps summed to ‚Çπ104,193 against a ‚Çπ100,000 input, because `risk` sizes each position independently off the risk budget with no built-in awareness of the running total. Fixed by tracking `capital_deployed` the same way `sector_deployed` already was, with a dedicated regression test (`TestAggregateCapitalCap`, 9 same-drift/distinct-sector symbols) ‚Äî synthetic tests with only 4 symbols hadn't summed past capital by chance, which is why this shipped past unit tests and was only caught live.

Positions below `min_ticket` (default ‚Çπ5,000) or with NaN/non-positive ATR (insufficient history ‚Äî never sized blind, the same discipline as the rest of the codebase) are excluded with a stated reason, never silently dropped or zero-filled. Each position carries a plain-English `rationale` string in the same evidence-ledger style as screen conditions (e.g. `"risk ‚Çπ1,000 √∑ stop distance ‚Çπ59.65 = 13 shares = ‚Çπ14,424 (14.4%)"`).

**API**: `POST /api/allocate` ‚Äî `{symbols, capital, method, risk_pct, max_positions, max_position_pct, sector_cap_pct, min_ticket, as_of, spec}` ‚Üí `{method, capital, positions, excluded, summary, baseline?, disclaimer}`. Logged to `data/allocation_log.jsonl` ‚Äî a sibling to `screen_log.jsonl` kept as its own file rather than interleaved, since the schemas genuinely differ (a position table, not a matched-symbol list) and `verify.check_screen_log`'s required-keys check would otherwise need to special-case allocation entries; if the caller passes the originating screen's `spec`, its `spec_hash` is logged too, tracing an allocation back to the screen that produced its symbol list.

**UI**: an "üí∞ allocate" button on the results panel opens a form (capital, a conservative/moderate/aggressive/custom risk-preset selector, method toggle, constraint fields) that sizes the current match list in its existing sort order; results render as a position table plus (for `risk`/`inverse_vol`) the equal-weight baseline table underneath for comparison, an excluded-with-reasons list, the disclaimer, and a CSV export. Resets when a new screen runs, the same discipline as the recent-screens panel.

## 12f. Screen backtester (v0.12 Item 14) ‚Äî event-study engine

`screener/backtest.py` answers: for any DSL spec, what happened after this signal fired historically, versus just holding the universe on the same dates? It is explicitly an **edge detector for filters, not a portfolio simulator** ‚Äî no position sizing, no compounding, no stops, no execution modelling (that's `allocate.py`'s job, applied once a screen has earned some trust here).

**Locked methodology.** Event = signal True at bar t AND no event for that symbol in the prior `cooldown` bars (default 20) ‚Äî de-dupes "signal stays true for a week" into one event, not near-duplicate samples. Entry at **open[t+1]** (bar-t close is where the signal fires; a close-t entry would be look-ahead); forward return at horizon h = close[t+h] / open[t+1] ‚àí 1. An event within h bars of its panel's end is excluded from *that horizon only*, never NaN-polluted into the stats for the others. Baseline is the same-date equal-weight mean forward return across all liquidity-passing universe symbols ‚Äî the part most home-built backtests get wrong, since it controls for the market regime on the signal date rather than comparing to an unconditional long-run average. Costs: gross and net (net = gross ‚àí a configurable round-trip %, default 0.30) always shown side by side. Statistics: per-horizon count/mean/median/hit-rate/p5/p95/worst-5%-mean on the excess distribution, a date-level block bootstrap (seeded, reproducible) on the event-date portfolio series ‚Äî never pooled-event t-stats, since same-date events across symbols are cross-correlated ‚Äî and an outright "insufficient events" suppression below 30 valid events at a horizon. An optional free-text `hypothesis` is logged with every run (multiple-testing is the real killer; writing the expectation down before seeing the answer is the cheapest available discipline). A survivorship caveat string is attached to every result, unconditionally ‚Äî the current-constituent universe flatters dip-buying/mean-reversion setups, since names that didn't survive to today aren't in it to drag the average down.

**Vectorized signal path.** Per-condition vectorizers (`_vec_*` functions in backtest.py) reproduce every cheap evaluator condition (compare/range/change/trend/cross/proximity/support_at_ma/volume_spike/gap/tight_range/flat_base/candle/rel_strength/sector, plus weekly-timeframe variants via an as-of expansion) as whole-panel pandas boolean Series, built to match `evaluator.py`'s per-bar semantics exactly (including edge cases like `support_at_ma`'s NaN-in-window override). The expensive set ‚Äî `near_support`, `near_resistance`, `breakout_resistance`, `bb_squeeze` (all O(lookback) pivot/percentile work per call) and the cross-sectional `rs_percentile`/`sector_rank`/`atr_pct_percentile` (each needing a universe-wide rank rebuild) ‚Äî instead sample the *existing* per-bar evaluator functions on a `stride`-bar date grid and forward-fill between samples, an explicit, documented approximation rather than a silent one.

**Acceptance test, not a runtime guarantee.** `verify_vectorizer_consistency()` samples random (symbol, date) pairs and asserts the vectorized signal matches `evaluate_symbol()` exactly. For cheap-only specs this holds at every date, so the test samples freely. For specs touching an expensive type, exactness is only guaranteed *at the stride-grid dates themselves* (the cross-sectional path literally reuses `cross_section.build_cross_section()`'s own cache at those dates, so it's the same computation twice by construction) ‚Äî so the test restricts its sample to grid-aligned dates for those specs, testing the guarantee actually made rather than a stronger one nobody promised.

**Perf reality vs. the original plan.** The spec's example stride (5 bars) was measured against the real 500-symbol/5-year store before shipping: `near_support`/`breakout_resistance`/`bb_squeeze` at stride=5 took roughly 6-15 minutes for a *single* condition ‚Äî well past any reasonable "<3min" budget. `stride=20` (~1 trading month between samples) measured at ~25-90s per expensive condition type and ~30-40s for a cross-sectional one, comfortably inside budget, and became the shipped default rather than the originally-planned 5 ‚Äî documented here rather than silently kept at a number that didn't hold up. A cheap-conditions-only spec runs in ~13-19s regardless of stride. The auto-detected one-at-a-time **sensitivity grid** (perturbs each numeric threshold ¬±2 steps, verdict "robust across range" vs. "edge concentrated at one value ‚Äî treat as curve-fit") fully reruns the signal path per variant, so it costs roughly (1 + 4√ón_params)√ó the core runtime ‚Äî for a spec with one sensitivity-eligible parameter and an expensive condition type, that can mean several minutes; it's exposed as an independent UI/API/CLI toggle (`sensitivity: bool`) for exactly that reason, not forced on.

**Cross-sectional history, simplified from the original plan.** Rather than a new vectorized date-indexed rank rebuild in `cross_section.py`, the shipped implementation calls the *existing* single-date `build_cross_section()` once per stride-grid date (not once per bar, and not a new code path) and forward-fills between samples the same way the per-symbol expensive types do. Measured fast enough in practice that the heavier rewrite wasn't needed; worth revisiting only if a future spec combines multiple distinct cross-sectional windows and this becomes the bottleneck.

**Two bugs found via live testing, fixed before shipping.** (1) The very first real `/api/backtest` call 500'd: `json.dumps()` rejects a raw Python `nan`, which a horizon-truncated event's row legitimately contains ‚Äî fixed with an explicit NaN‚Üí`None` pass over the events table (`_events_to_records`) before it leaves `backtest_spec`. (2) This item's own test suite ‚Äî which builds and discards many short-lived synthetic `panels` dicts ‚Äî started intermittently corrupting *unrelated* preset tests elsewhere in the same pytest run. Root cause: `cross_section.py`'s in-process cache keyed on `id(panels)` alone, a risk its own docstring already named ("a GC-reused id could otherwise serve stale ranks indefinitely") but that no prior test suite had exercised heavily enough to trigger. Fixed by adding `frozenset(panels)` (the symbol set) to the cache key, with a regression test that plants a stale entry under a real id() with a different symbol set and confirms the genuine computation wins rather than the stale hit.

**API + UI + CLI.** `POST /api/backtest` ‚Äî `{spec, horizons, cooldown, cost_pct, min_turnover_cr, stride, min_events, hypothesis, sensitivity}` ‚Üí per-horizon stats, the per-event table, an event timeline, the sensitivity grid, and the survivorship note; logged to a new `data/backtest_log.jsonl` (spec hash + params + hypothesis + per-horizon summary ‚Äî not the full per-event table, which alone can run to thousands of rows). The UI's "üß™ backtest" panel (opened from the results view, same pattern as "üí∞ allocate") replays the current spec: a form for all the run parameters, per-horizon tables (gross/net excess, hit rate, p5/p95, worst-5%, the bootstrap CI), an excess-return histogram vs. baseline, an events-per-month bar chart ("a signal that stopped firing is a finding"), the sensitivity table with its verdict line, the survivorship banner, and a CSV export of the per-event table. `python -m screener.cli backtest "<query>" [--preset id] [--json] [--horizons ...] [--cooldown N] [--cost-pct X] [--stride N] [--hypothesis "..."] [--no-sensitivity]` gives the same engine a readable text report. All three verified live against the real 500-symbol store; the UI additionally verified via Playwright (zero console errors, correct rendering with the sensitivity grid on and off, print-mode visibility).

**Deferred, not struck**: closing the loop by adding each of the 26 existing presets' own backtest summary to its `evidence` object. This is a distinct, sizeable follow-up (running and curating 26 backtests, settling on consistent robust/fragile wording) rather than a natural extension of shipping the engine ‚Äî revisit as its own scoped effort.

## 12g. Universe registry (v0.13 Item 15 Phase A)

`screener/universes.py` turns "which symbols, which price store, which benchmark" into a config entry instead of hardcoded paths scattered through `config.py`, `data_ingest.py`, `universe.py`, `cli.py`, and `webapp.py`. ROADMAP Item 15 (equity depth track, v0.13) landed in two slices: a foundation refactor (2026-07-09, zero behaviour change, `nifty500` only, full details of that slice below) followed by onboarding a real second universe, `nse_full` (2026-07-10). Multi-asset phases (FX, crypto) were separately descoped 2026-07-07 in the same ROADMAP item: equity only, other asset classes get their own engines later rather than a shared field-mask/calendar abstraction.

**Registry.** `Universe` is a frozen dataclass: `{id, name, benchmark_ticker, liquidity_gate_cr, survivorship_note}`. `nifty500` (‚Çπ0.5cr gate) and `nse_full` (all 2,047 NSE EQ-series symbols per NSE's own equity listing ‚Äî vs. nifty500's index-membership list ‚Äî ‚Çπ2cr gate: a much longer tail of thin names needs a stricter floor). `liquidity_gate_cr` is the first field that earned its place once a second universe genuinely needed to differ on it ‚Äî during the foundation slice, with only one universe registered, `Universe` deliberately had no such field (`config.MIN_MEDIAN_TURNOVER_CR` stayed global). `sector_enabled` is still not a field: NSE's raw equity listing carries no sector/industry classification at all (that's an index-methodology concept the Nifty 500 CSV has and the raw listing doesn't), so `industry` is left empty for nse_full and `sector`/`sector_rank` conditions simply find nothing to match ‚Äî the existing NaN-industry handling already degrades gracefully, no registry flag needed to make that correct.

**Symbol-list fetch, dispatched per universe.** `universe.py`'s `fetch_universe()` now dispatches to a per-`universe_id` fetcher (`_FETCHERS` dict) rather than one hardcoded format: `_fetch_nifty500` (unchanged) parses the Nifty 500 index CSV (symbol/company_name/industry); `_fetch_nse_full` parses NSE's `EQUITY_L.csv` full listing (symbol/name only, filtered to `SERIES == "EQ"`, same convention `bhavcopy.py` already used), with the same fetch-live-else-fall-back-to-cache resilience nifty500's fetcher has always had.

**Per-universe storage.** Every universe gets `data/{universe_id}/{prices.parquet, universe.csv, benchmark.parquet}`. `config.py` gained `price_store()`/`universe_file()`/`benchmark_store()`/`liquidity_gate_cr()` functions, each taking an optional `universe_id` (default `universes.DEFAULT_UNIVERSE`). For the default universe specifically, each function returns the corresponding module-level attribute/constant (`config.PRICE_STORE`, `config.MIN_MEDIAN_TURNOVER_CR`, etc.) rather than recomputing ‚Äî this is why those attributes still exist as plain values rather than being replaced outright: several existing tests do `monkeypatch.setattr(config, "PRICE_STORE", ...)`, and `config_local.toml`'s `MIN_MEDIAN_TURNOVER_CR` override must keep working for nifty500 specifically; this indirection keeps both working unchanged while nse_full gets its own fixed registry value.

**Migration.** Before this item, the one (implicit) universe's files sat flat in `data/` ‚Äî `prices.parquet`, `nifty500.csv`, `benchmark.parquet`. `config._migrate_legacy_nifty500_layout()` runs once at import time: if the legacy flat files exist and the new per-universe location doesn't, it moves them (`Path.rename`, atomic on POSIX) and prints a one-line notice to stderr; otherwise it's a no-op. Verified on the real dev-machine store ‚Äî confirmed the actual 500-symbol files moved into `data/nifty500/` correctly, and that CLI screens and the webapp produced identical results (same match counts, same `as_of`, `panel_count: 500`) before and after.

**The real nse_full backfill.** `python -m screener.cli backfill --universe nse_full` ran the actual ~2,047-symbol yfinance backfill live: 2,065,698 rows, 2021-07-12 ‚Üí 2026-07-10, in ~15 minutes ‚Äî comfortably under the 30-60 minute estimate made before checking (the bhavcopy store, the original plan's intended nse_full source, turned out to have only 7 days of history since side-by-side collection had just started, nowhere near the 5 years indicators need; reusing the existing yfinance pipeline with a bigger symbol list sidestepped needing a new bulk-historical-backfill mechanism entirely, and inherits yfinance's `auto_adjust` correctness rather than needing bhavcopy's separate corporate-action adjustment pipeline exercised at 4x scale).

**Memory gate ‚Äî measured, passes, no rearchitecture needed.** With both universes' panels resident simultaneously in one process (500 nifty500 + 1,956 nse_full ‚Äî 91 of nse_full's 2,047 symbols dropped by the existing 60-bar-minimum-history filter, same as any thin-history name), peak RSS was 782 MB, comfortably under the 4 GB target. The on-demand/LRU panel-build fallback the ROADMAP spec allowed for turned out to be unnecessary ‚Äî the simple "keep everything resident" architecture nifty500 already used just works at this scale.

**Webapp universe selector.** `GET /api/universes` (registered universes + which is active) and `POST /api/universe` (switches a single process-wide active universe, stored in a module-level `_ACTIVE_UNIVERSE`). Deliberately process-wide rather than a per-request field on every endpoint: this is a local, single-user tool, not a multi-tenant service, so "which universe is the UI currently showing" is one selection, not something that needs threading through every request body. Every endpoint already read through `_load_state(universe_id)` from the foundation slice, so switching the 8 call sites from `_load_state()` to `_load_state(_ACTIVE_UNIVERSE)` was the entire plumbing change. UI: a header `<select>`, populated from `/api/universes`, hidden via `loadUniverses()` when fewer than 2 universes are registered (dead UI otherwise). Switching resets any in-progress screen/allocation/backtest (a prior universe's results don't apply to the new one) and shows an honest busy message ‚Äî the first load of a not-yet-cached universe in a server session has a real cost (nse_full: ~165s cold, confirmed live), cached instantly after that (also confirmed live: repeat switches between already-loaded universes complete in well under 100ms).

**Bug found via live testing, fixed before shipping.** The very first nse_full backtest printed `backtest.py`'s hardcoded, nifty500-worded survivorship caveat ("Nifty 500 constituent list‚Ä¶") regardless of which universe was actually active ‚Äî simply wrong for a 2,047-symbol universe with even heavier churn than the index. Fixed by adding a `survivorship_note: str | None = None` parameter to `backtest_spec()` (defaults to the existing module constant for backward compatibility; CLI and the webapp backtest endpoint both now pass `universes.get(universe_id).survivorship_note` explicitly). A regression test asserts the webapp's backtest response carries nse_full's own wording, not nifty500's, when nse_full is the active universe.

**Explicitly not done this pass**: `nse_etf`, preset `universes` tags (no second universe existed to differentiate presets against when originally scoped; revisit once `nse_etf`'s sector-disabled nature makes some presets genuinely inapplicable), and the fuller original registry schema (`symbol_source` as a stored callable rather than a `universe.py`-internal dispatch dict).

**Verification.** 24 new tests total across the two slices (`tests/test_universes.py`: registry lookup/errors, path-helper and liquidity-gate defaulting (including that nifty500's `config_local.toml` override still works), migration idempotency, per-universe symbol fetch ‚Äî nse_full's EQ-series filtering mocked, its cache-fallback-on-network-failure path tested directly ‚Äî and CLI `--universe` acceptance/rejection; `tests/test_evaluator.py::TestUniverseEndpoints`: the webapp selector's list/switch/422 contract and the survivorship-note regression, with the switch test pointed at a `tmp_path` so it doesn't depend on nse_full's now-real data on this dev machine). Full suite (236 tests) green throughout. Live-verified end to end via Playwright: switch to nse_full, run a screen (514 matches, correct liquidity exclusions, industry column correctly empty), switch back to nifty500 ‚Äî zero console errors.
